## 최종 톤 선언

### 한 줄 정의

- `노션식 여백 + 한국어 가독성 + 운영 도구의 신뢰감 + 과장 없는 SaaS 워크스페이스`

### 유지할 인상

- 차분함
- 신뢰감
- 문서형 작업 화면
- 행동이 먼저 보이는 절제감

### 피할 인상

- 행정툴 같은 과한 박스 구획
- 스타트업 랜딩페이지 같은 과장된 장식
- 보라색 AI 제품 감성
- 설명문이 많은 매뉴얼형 UI
- 카드 안의 카드가 겹친 복잡한 대시보드

## 핵심 디자인 원칙

### 1. 여백이 박스보다 먼저 읽혀야 한다

- 구조 구분의 기본 수단은 `padding`, `gap`, `alignment`다.
- 보더와 배경은 꼭 필요할 때만 쓴다.

### 2. 1차 정보는 화면마다 1개 묶음만 강하게

- 첫 진입: 업로드/녹음 CTA
- 생성 흐름: 선택한 오디오와 저장
- 선택 후: 원문 전체

### 3. 카드 사용은 최소화한다

- 바깥 레벨 주요 패널은 허용
- 내부 정보는 가능한 한 카드가 아니라 섹션과 여백으로 나눈다
- 한 화면에서 `강한 카드`는 최대 2종류만 허용

### 4. 설명보다 구조로 이해되게 한다

- 긴 문장은 없애고
- 제목, 소제목, 액션 레이블, 최소 메타로 흐름을 읽히게 한다

### 5. 따뜻하지만 채도는 낮게

- 현재 웜 뉴트럴 감성은 유지한다
- 기존 주황 포인트는 더 눌러서 `조용한 구리빛 accent`로 쓴다

## 타이포그래피

## 기본 결정

- 워크스페이스 전용 폰트 교체는 하지 않는다.
- 현재 전역 폰트인 `Noto Sans KR`를 유지한다.
- 이유
  - 이미 글로벌에 적용돼 있다
  - 한국어 상담 원문 가독성이 우선이다
  - 이번 차수의 목적은 폰트 교체보다 정보 위계 정리다

## 타이포 역할

| 역할          | 크기    | 굵기    | 사용 규칙           |
| ------------- | ------- | ------- | ------------------- |
| 페이지 타이틀 | 32-36px | 700     | 헤더에서 1회만      |
| 섹션 타이틀   | 22-28px | 700     | 좌측/중앙 주요 섹션 |
| 기록 제목     | 18-22px | 600-700 | 리스트/상세 공통    |
| 본문 텍스트   | 14-16px | 400-500 | 기본 가독성         |
| 보조 텍스트   | 12-13px | 400-500 | 메타, 힌트, 상태    |
| 액션 버튼     | 14-15px | 600     | 동사 시작           |

## 타이포 규칙

- 한국어 서비스이므로 과한 영문 uppercase eyebrow를 줄인다.
- `YEON 상담 기록` 같은 보조 라벨은 허용하되, 화면마다 반복되는 uppercase 메타 라벨은 최소화한다.
- 원문 텍스트는 가장 읽기 쉬운 줄 간격을 우선한다.
- 원문 외 영역은 `짧고 단단한 문장`을 사용한다.

- 상태색은 정보 전달용으로만 제한

## 색상 사용 규칙

- `accent`는 CTA, 활성 탭, 현재 선택 기록 정도에만 쓴다.
- 메트릭 카드나 배지 전체를 강한 색으로 칠하지 않는다.
- 상태는 색만으로 전달하지 않고 텍스트/아이콘을 함께 둔다.
- 배경 그라데이션은 페이지 전체에서 아주 약하게만 허용한다.
- 내부 카드에는 그라데이션을 기본값으로 두지 않는다.

- 현재 화면은 요소 수보다 패널 수가 많아 답답하다.
- 앞으로는 같은 정보를 더 작은 카드로 쪼개지 말고 섹션 간격으로 해결한다.
- 첫 화면 CTA 주변은 다른 영역보다 여백을 더 크게 둔다.

## 보더, 반경, 그림자

## 반경

| 용도           | 값      |
| -------------- | ------- |
| 주요 외곽 패널 | 20px    |
| 내부 약한 블록 | 16px    |
| 입력창/버튼    | 14-16px |
| 칩/배지        | 999px   |

## 보더

- 기본 보더는 `1px solid var(--border-soft)`
- 실제 구분이 필요할 때만 `var(--border-default)` 사용
- 점선 보더는 빈 상태 1곳 정도에서만 허용

## 그림자

| 용도       | 값                                   |
| ---------- | ------------------------------------ |
| 기본       | `0 8px 24px rgba(30, 27, 24, 0.05)`  |
| hover/선택 | `0 12px 28px rgba(30, 27, 24, 0.08)` |
| 없음       | 내부 블록 기본값                     |

## 적용 원칙

- 현재 32px 라운드와 큰 그림자는 무겁다.
- 외곽 패널만 아주 약한 그림자를 쓰고, 내부 블록은 그림자 없이 처리한다.
- `카드 안 카드` 구조에는 그림자를 절대 중첩하지 않는다.

## 배경과 표면

### 페이지 배경

- 단색에 가까운 웜 뉴트럴을 기본으로 한다.
- 상단 radial highlight는 있더라도 매우 약하게 줄인다.
- 화면의 인상은 `배경 효과`가 아니라 `콘텐츠 여백`에서 나와야 한다.

### 표면 계층

| 계층     | 사용처                                          |
| -------- | ----------------------------------------------- |
| 캔버스   | 페이지 전체                                     |
| 1차 표면 | 좌측/중앙/우측 주요 패널                        |
| 2차 표면 | 접힘 영역, 선택 오디오 미리보기, 약한 상태 블록 |
| 무배경   | 요약/액션/원문 내부 구획                        |

## 컴포넌트 규칙

## 상단 헤더

- 제목 + 1줄 보조 문구 + 약한 보조 액션
- 메트릭 카드 금지
- 헤더 자체는 `정보 보드`가 아니라 `문서 제목줄`처럼 보여야 한다

## 생성 CTA 블록

- `파일 업로드`, `브라우저 녹음` 2개를 가장 먼저 배치
- 같은 크기 또는 거의 같은 위계
- 설명문은 1줄까지만 허용
- 추가 입력은 접힘 영역 뒤

## 기록 리스트

- 각 행은 하나의 카드라기보다 `조용한 선택 행`에 가깝게
- 보여줄 것
  - 학생명
  - 기록 제목
  - 날짜/길이
  - 상태
- 태그와 미리보기는 약하게, 필요할 때만

## 상세 헤더

- 기록 제목
- 핵심 상태
- 최소 메타 2~3개
- 긴 설명문 금지

## 원문 뷰어

- 가장 넓은 영역
- 검색창은 있지만 과장하지 않음
- 세그먼트는 문서처럼 읽히게 하고, 시커먼 배경이나 과한 분리선은 쓰지 않는다

## 요약 블록

- 큰 카드 4장보다 `짧은 구조화 블록`이 낫다
- `핵심 상담 내용 / 학생 문제 포인트 / 보호자 요청사항 / 다음 액션`
- 각 블록은 제목 + 2~3개 포인트 수준

## AI 패널

- 선택 후에만 등장
- 제목, 입력창, 최근 대화, 퀵프롬프트 일부만 우선 노출
- 컨텍스트 설명은 접힘 처리
- 채팅 버블은 과하게 둥글거나 장식적이면 안 된다

## 문구 시스템

## 문구 길이 규칙

- 제목: 12자 안팎
- 보조 문구: 1줄
- 상태 문구: 1문장
- 설명이 길어지면 문구가 아니라 구조 문제로 본다

## 문구 톤

- 감성적 수식 금지
- 실무형, 동사형, 결과 중심
- 예
  - `파일 업로드`
  - `브라우저 녹음`
  - `원문 확인`
  - `요약 정리`
  - `후속 조치`

## 금지 패턴

- `기록을 선택해 원문을 읽고...` 같은 절차 설명 장문
- `AI 답변 방식`, `선택 가능 기록` 같은 운영 메트릭 제목
- 가치 제안 문장을 화면마다 반복

## 구현 우선순위

### 먼저 바꿀 것

3. 선택 전 AI 패널 제거
4. 카드 수와 보더/그림자 강도 축소

### 나중에 바꿀 것

1. 앱 전체 폰트 변경

## 구현 시 메모

- `Noto Sans KR` 유지가 기본값이다.
- Tailwind 기본 scale을 막지 않는다.
- 전역 CSS는 건드리지 않고, 워크스페이스 범위에서 먼저 톤을 정리한다.
- 21st 레퍼런스는 현재 API key 부재 시 생략 가능하지만, 실제 구현 전에 키가 생기면 inspiration 단계로 한 번 더 교차 검토하는 것이 좋다.

## 차수 5 입력값

- 차수 5에서는 이 디자인 시스템 메모를 기준으로 `상단 헤더`, `생성 진입`, `생성 폼`, `기록 탐색`, `선택 전 빈 상태`, `선택 후 본문`, `AI 패널`의 책임 분리 설계를 작성한다.
