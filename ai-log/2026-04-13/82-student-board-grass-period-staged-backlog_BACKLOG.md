# 82. 출석·과제 잔디형 이력 + 스페이스 진행기간 단계별 백로그

## 공통 목표

- 출석·과제 이력을 긴 날짜 리스트가 아니라 `진행기간 기반 잔디형 블럭`으로 바꾼다.
- 학생 상세는 `개요`가 아니라 기본 시스템 탭 `출석·과제`에서 상세 이력을 본다.
- 학생 상세 상단 헤더 카드에 진행기간 기반 잔디 요약을 붙인다.
- 진행기간을 먼저 source of truth로 세운 뒤, 그 기간을 기준으로 보드와 학생 상세를 렌더링한다.
- 구현은 작은 landing 단위로 잘라서 각 차수가 `간단한 작업`처럼 느껴지게 만든다.

## 공통 전제

- 현재 들어가 있는 날짜별 리스트 중심 구현은 최종 UX가 아니므로 다음 차수에서 그대로 확장하지 않는다.
- 한 차수에 `데이터 모델 + 탭 구조 + 대형 UI`를 동시에 섞지 않는다.
- 잔디 셀은 하루당 `출석/과제 분할 블럭`으로 표현한다.
- `unknown`은 완전 빈칸이 아니라 `약한 회색 칸`으로 표현한다.
- 진행기간 규칙은 `시작일 고정`, `종료일 연장만 허용`, `종료일 단축 불가`다.
- import 진행기간은 `스페이스별 개별 입력`으로 간다.

## 차수 1. 진행기간 source of truth 정리

### 작업내용

- 스페이스의 `startDate`, `endDate`를 출석·과제 화면이 신뢰할 수 있는 공식 필드로 재확인한다.
- `시작일 변경 불가 / 종료일 연장만 허용` 규칙을 API와 UI에 반영한다.
- `빈 스페이스 만들기` 흐름에서 진행기간 입력을 노출한다.
- `가져오기 분석 결과`에서 각 스페이스(코호트) 행에 진행기간 입력 UI를 추가한다.
- 학생관리 스페이스 헤더에 진행기간 표시를 추가한다.
- 기간이 비어 있는 기존 스페이스에 대한 fallback 문구를 정한다.

### 논의 필요

- 진행기간이 필수인지 선택인지
- 기존 스페이스에 기간이 없을 때 강제 입력으로 전환할지
- 이미 생성된 스페이스에서 `종료일 연장` UI를 어디에 둘지

### 선택지

- 선택지 A: 생성 입력 + 종료일 연장 UI까지 이번 차수에 같이 연다
- 선택지 B: 생성 입력만 이번 차수에 넣고 연장은 다음 차수로 미룬다

### 추천

- `선택지 A`
- 이유: 시작일/종료일 규칙이 이 차수에서 안 닫히면 잔디 렌더 기간이 계속 흔들린다.

## 차수 2. 학생 상세 정보구조 재편

### 작업내용

- 기본 시스템 탭에 `출석·과제`를 `개요` 바로 다음 순서로 추가한다.
- `개요`에 임시로 들어간 출석·과제 이력 UI를 제거한다.
- 학생 상세 라우팅/탭 상태에서 새 시스템 탭을 기본 탭처럼 취급하도록 연결한다.
- 학생 상세 상단 헤더 카드에 잔디가 들어갈 자리와 요약 메타 슬롯을 설계한다.

### 논의 필요

- 헤더 잔디 아래에 작은 범례를 둘지 말지
- 탭 라벨을 `출석·과제` 그대로 둘지 더 짧게 줄일지

### 선택지

- 선택지 A: 작은 범례를 둔다
- 선택지 B: hover 툴팁만 두고 범례는 생략한다

### 추천

- `선택지 B`
- 이유: 헤더 공간은 이미 빡빡하고, 범례는 보드에서 더 중요하다.

### 사용자 방향

## 차수 3. 출석·과제 이력 모델을 잔디 친화형으로 재정리

### 작업내용

- 현재 `날짜 리스트용` 응답 형태를 재검토하고, `기간 내 일자 배열 + 셀 상태` 중심 DTO를 정의한다.
- 수동 수정과 공개 체크인이 모두 같은 이력 source에 누적되도록 write path를 고정한다.
- 보드용 집계 API와 학생 상세용 집계 API를 분리해 응답 크기를 제어한다.
- 하루 셀에서 `출석`, `과제`가 각각 어떤 상태 값을 가지는지 최종 상수 집합을 정리한다.
- `unknown`은 약한 회색 칸으로 읽히도록 시각 규칙을 상수화한다.

### 논의 필요

- 셀 색 표현을 얼마나 강하게 대비시킬지
- assignment link 같은 상세 데이터는 집계 응답에서 빼고 세부 리스트 API로만 둘지

### 선택지

- 선택지 A: 잔디 집계 응답과 상세 리스트 응답을 분리
- 선택지 B: 하나의 큰 응답에 집계와 세부를 모두 포함

### 추천

- `선택지 A`
- 이유: 보드와 학생 상세 헤더는 고밀도 집계만 빠르게 받고, 상세 탭만 별도 리스트를 받는 편이 성능과 구조가 낫다.

### 사용자 방향

## 차수 4. 출석·과제 보드 운영 화면을 고밀도로 재설계

### 작업내용

- 상단 카드를 `전체 학생 / 출석 / 과제 완료` 3개만 콤팩트하게 남긴다.
- 학생별로 한 줄씩 보이는 `진행기간 잔디 행`을 만든다.
- 각 날짜 셀은 `출석/과제 분할 블럭`으로 렌더한다.
- 기간이 길어질 때 가로 스크롤, sticky 이름 열, 범례, hover 상세를 정리한다.

### 논의 필요

- 보드를 주 단위 묶음으로 구분할지, 월 구분선만 둘지
- 모바일에서는 같은 밀도를 유지할지, 일부 축약할지

### 선택지

- 선택지 A: 주 단위 컬럼 그룹 + 월 라벨
- 선택지 B: 월 구분선만 있는 단순 연속 그리드

### 추천

- `선택지 A`
- 이유: GitHub 잔디 감각을 유지하면서도 운영자가 날짜 맥락을 잃지 않는다.

### 사용자 방향

## 차수 5. 학생 상세 헤더 잔디 요약

### 작업내용

- 학생 이름/전화번호/이메일이 있는 현재 헤더 카드 자체를 확장해 잔디를 붙인다.
- 잔디는 보드와 같은 셀 규칙을 쓰되 학생 1명 전용으로 더 촘촘하게 보여준다.
- 출석률, 과제 완료율 같은 한 줄 요약 메타를 헤더와 함께 정리한다.
- hover/focus 시 해당 날짜의 간단 상태를 노출한다.

### 논의 필요

- 헤더 잔디를 아래 전체폭으로 깔지, 내부 2열 레이아웃으로 둘지
- 진행기간이 길 때 월 라벨을 헤더에도 노출할지

### 선택지

- 선택지 A: 헤더 아래 전체폭 잔디
- 선택지 B: 헤더 내부 2열 레이아웃 잔디

### 추천

- `선택지 A`
- 이유: 작은 카드 옆에 무리하게 넣으면 가독성이 깨지고 모바일 대응이 어려워진다.

## 차수 6. `출석·과제` 탭의 상세 이력 화면

### 작업내용

- 새 시스템 탭 `출석·과제`에서 날짜순 상세 리스트를 보여준다.
- 리스트에는 날짜, 출석 상태, 과제 상태, 링크, 수정 주체, 수정 시각을 담는다.
- 헤더 잔디와 탭 상세가 같은 source를 바라보도록 검증한다.
- 필요하면 날짜 필터와 간단 검색을 넣는다.
- 1차 차수에서는 `조회 전용`으로 고정하고 수정 액션은 넣지 않는다.

### 논의 필요

- 기본 정렬을 최신순으로만 둘지, 월별 그룹핑을 줄지
- 상세 이력의 hover 정보량을 어디까지 둘지

### 선택지

- 선택지 A: 읽기 중심 리스트로 먼저 출시
- 선택지 B: 리스트 안에서 바로 수정까지 허용

### 추천

- `선택지 A`
- 이유: 수정 액션까지 넣으면 상태 전이가 커져서 차수가 커진다. 먼저 조회 구조를 닫는 편이 안전하다.
