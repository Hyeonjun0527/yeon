# AI 분석 결과 JSON 구조화 및 시스템 자동 반영 백로그

## 배경

현재 녹음본 업로드 → AI 분석 → 결과가 채팅 패널에 마크다운으로 표시된다.
이 방식은 데이터가 휘발되고, 수강생 연동이 안 되고, 후속 조치 추적이 불가능하다.

**방향 전환**: AI 분석 결과를 JSON 구조로 받아서 시스템이 자동으로 레코드에 저장하고,
수강생 매칭/타임라인 기록/후속 조치 생성까지 자동화한다.
채팅 패널은 분석 결과 출력용이 아니라 후속 질문용으로만 남긴다.

> 기존 `6-counseling-analysis-pipeline-upgrade_BACKLOG.md`의 1차 "마크다운 유지" 추천을 폐기하고 JSON 구조화(B안)로 전환한다.

## 1차: AI 분석 응답 JSON 스키마 정의 + 프롬프트 변경

### 작업내용

- AI 분석 응답 JSON 스키마 정의 (TypeScript 타입 + Zod 검증)
  ```ts
  interface AnalysisResult {
    summary: string;
    member: {
      name: string | null;
      traits: string[];
      emotion: string;
    };
    issues: Array<{
      title: string;
      detail: string;
      timestamp?: string;
    }>;
    actions: {
      mentor: string[];
      member: string[];
      nextSession: string[];
    };
    keywords: string[];
  }
  ```
- `buildInitialAnalysisPrompt()` 변경: JSON 응답 강제, 용어 규칙 반영 (수강생/멘토)
- AI 응답 파싱 + Zod 검증 + 실패 시 재시도 로직
- 기존 마크다운 스트리밍 채팅 출력 제거

### 논의 필요

- 스트리밍 JSON 파싱이 필요한가, 완성된 응답을 한번에 받을 것인가
- OpenAI structured output (response_format: json_schema) 사용 여부

### 선택지

- A: `response_format: { type: "json_object" }` 사용 — 간단하지만 스키마 강제력 약함
- B: `response_format: { type: "json_schema", json_schema: ... }` 사용 — 스키마 완전 강제
- C: 마크다운 응답 받고 후처리로 JSON 추출 — 기존 구조 변경 최소

### 추천

B — structured output으로 스키마를 강제하면 파싱 실패 가능성이 거의 없음. 스트리밍은 불필요 (분석 결과는 한번에 저장하면 됨)

## 2차: 분석 결과 DB 저장 + 레코드 상세 UI

### 작업내용

- `counseling_records` 테이블에 `analysis_result` (JSONB) 컬럼 추가 또는 별도 테이블
- 분석 완료 시 JSON 결과를 레코드에 자동 저장
- 레코드 상세 패널 UI 변경:
  - 핵심 요약 카드
  - 수강생 정보 카드 (이름, 특징, 감정)
  - 주요 이슈 리스트 (타임스탬프 클릭 시 해당 구간 재생)
  - 후속 조치 체크리스트 (멘토 액션 / 수강생 과제 / 다음 상담)
- 채팅 패널에서 분석 결과 표시 제거 → "분석 완료" 토스트만 표시

### 논의 필요

- analysis_result를 레코드 테이블 컬럼으로 넣을지, 별도 테이블로 분리할지
- 재분석 시 이전 결과 이력을 남길지

### 선택지

- A: 레코드 테이블에 JSONB 컬럼 — 단순, 1:1 관계
- B: 별도 analysis_results 테이블 — 이력 관리 가능, 재분석 히스토리

### 추천

A — 현재 단계에서 분석 이력이 필요한 시나리오가 없음. 나중에 필요하면 마이그레이션

## 3차: 수강생 자동 매칭 + 타임라인 연동

### 작업내용

- 분석 결과의 `member.name`으로 기존 수강생 퍼지 매칭
  - 정확히 일치 → 자동 연결
  - 유사 매칭 → 멘토에게 확인 UI ("민지" → "김민지 맞나요?")
  - 미매칭 → 새 수강생 등록 제안
- 매칭된 수강생의 타임라인에 상담 기록 자동 추가
- 수강생 상세 화면에서 상담 이력 조회 가능

### 논의 필요

- 퍼지 매칭 기준 (동명이인 처리)
- 같은 스페이스 내에서만 매칭할지

### 선택지

- A: 같은 스페이스 내 이름 완전 일치만 자동 매칭 — 안전하지만 수동 연결 빈도 높음
- B: 스페이스 내 유사도 매칭 + 확인 UI — 편의성 높지만 구현 복잡

### 추천

A로 시작 — 완전 일치 자동 매칭 + 나머지는 수동 선택 드롭다운. 단순하고 오매칭 위험 없음

### 사용자 방향

## 4차: 후속 조치 시스템 (할 일 / 체크리스트)

### 작업내용

- `actions` 테이블 신설: 레코드 연결, 담당자, 상태(미완료/완료), 카테고리(mentor/member/nextSession)
- 분석 완료 시 `actions.mentor`, `actions.member`, `actions.nextSession` 항목 자동 생성
- 레코드 상세에서 체크리스트 UI (토글로 완료 처리)
- 멘토 대시보드에 "미완료 액션" 위젯

### 논의 필요

- 수강생에게 과제를 알림으로 보낼 것인지 (현재 수강생 앱 없으므로 보류)
- 액션 기한 설정 여부

### 선택지

- A: 단순 체크리스트 (완료/미완료) — MVP
- B: 기한 + 알림 + 우선순위 — 풀 태스크 관리

### 추천

A — MVP로 체크리스트만. 태스크 관리 도구를 만드는 게 아니라 상담 후속 조치 추적이 목적

## 5차: 채팅 패널 역할 재정의

### 작업내용

- 채팅 패널을 "분석 결과 기반 후속 질문" 용도로 전환
- 분석 결과 JSON을 채팅 컨텍스트에 주입하여 "민지의 출석률은?", "이전 상담과 비교해줘" 등 질의 가능
- 분석 결과를 채팅에 직접 출력하는 기존 로직 완전 제거

### 논의 필요

- 채팅 컨텍스트에 분석 JSON + 전사 원문을 모두 넣을지, 분석 JSON만 넣을지

### 선택지

- A: 분석 JSON만 컨텍스트 — 토큰 절약
- B: 분석 JSON + 전사 원문 — 원문 기반 질의 가능

### 추천

B — 전사 원문이 있어야 "00:43 부분에서 뭐라고 했어?" 같은 질문에 답할 수 있음
