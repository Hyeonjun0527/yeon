# 상담 기록 실전 STT 워크플로우 구현 BACKLOG

## 차수 1

### 작업내용

- 상담 음성 파일을 실제로 저장할 수 있는 업로드 경로와 영속 스키마를 추가한다.
- 음성 파일 메타데이터, 전사 상태, 원문 텍스트, 세그먼트를 DB에 저장하는 구조를 만든다.
- 로컬 파일 저장소를 구성해 업로드 후에도 원본 음성을 다시 열 수 있게 한다.

### 논의 필요

- 배포 환경에서 음성 파일을 컨테이너 로컬 디스크에 둘지, 별도 object storage로 옮길지 후속 결정이 필요하다.
- 이번 차수는 현재 저장소 현실에 맞춰 로컬 영속 디렉터리를 기본값으로 두되, 이후 외부 스토리지로 바꾸기 쉬운 경계를 유지해야 한다.

### 선택지

- `A. DB만 저장하고 파일은 즉시 전사 후 폐기`
- `B. 음성 파일과 전사 결과를 함께 영속 저장`

### 추천

- `B`를 추천한다.
- 이유는 전사 실패 재시도, 품질 검토, 원본 재확인이 가능해야 실제 운영 기능으로 쓸 수 있기 때문이다.

### 사용자 방향

## 차수 2

### 작업내용

- 한국어 STT를 실제로 수행하는 서버 API를 구현한다.
- 업로드 성공 후 전사 완료 또는 실패 상태가 기록에 남게 한다.
- 실패 시 에러 상태를 저장하고, 저장된 원본 음성으로 재전사할 수 있게 만든다.

### 논의 필요

- 대용량 파일의 비동기 처리 큐까지 이번 차수에 넣을지 여부는 후속 검토가 필요하다.
- 이번 구현은 `저장 먼저 -> 전사 -> 실패 시 재시도` 흐름으로 실제 사용 가능한 수준을 우선 만든다.

### 선택지

- `A. 업로드와 전사를 하나의 요청에서 처리`
- `B. 업로드와 전사를 완전히 분리한 작업 큐 기반`

### 추천

- `A`를 추천하되, 저장된 음성 파일 기준 재전사 API를 별도로 두어 실패 복구성을 보강한다.
- 이유는 현재 저장소 단계에서 과한 작업 큐보다 실제 동작하는 end-to-end 흐름을 먼저 만드는 편이 낫기 때문이다.

### 사용자 방향

## 차수 3

### 작업내용

- 상담 기록 워크스페이스를 실제 API 기반으로 전환한다.
- 브라우저 녹음, 파일 업로드, 상태 표시, 원문 검색, 긴 원문 가독성을 실제 제품 UX로 다듬는다.
- 샘플 데이터 기반 가짜 상태를 제거하고 실데이터만 보여주도록 정리한다.

### 논의 필요

- AI 채팅 패널은 현재 핵심 범위 밖이므로, 가짜 답변을 유지하지 않고 실제 기능 범위에 맞게 정리할 필요가 있다.

### 선택지

- `A. 기존 가짜 AI 패널 유지`
- `B. 실제 음성/원문/상태 기능에 집중하도록 패널을 재구성`

### 추천

- `B`를 추천한다.
- 이유는 핵심 코어를 구현하는 단계에서 가짜 분석 패널은 제품 신뢰를 오히려 해치기 때문이다.
