# 학생관리 CRM 라운드 2 제품형 대시보드 보강 백로그

## 1차

- 작업내용
  - 라운드 2 KPI 브리핑을 실제 제품 홈 상단 구조로 재배치한다.
  - 설명형 공모전 섹션 대신 학생 큐, 학생 상세, 오늘 액션 보드가 먼저 보이는 대시보드 홈을 연결한다.
  - 전역 색상, 표면, 경계, 텍스트 위계를 CSS 변수로 정리해 일관된 디자인 시스템을 만든다.
- 논의 필요
  - 홈에서 공모전 설명 섹션을 유지할지, 제품 대시보드만 남길지 판단이 필요하다.
- 선택지
  - A. 제품 대시보드 우선으로 전환하고 설명형 섹션은 제거한다.
  - B. 상단만 제품화하고 하단 설명 섹션은 계속 유지한다.
- 추천
  - A. 사용자가 요구한 학생관리 MVP와 round 2의 first screen 목표에 더 직접적으로 맞는다.
- 사용자 방향

## 2차

- 작업내용
  - 상단 KPI는 `오늘 케어 학생 수`, `개입 대기`, `반복 개념`, `오늘 액션`의 4개 고정 항목으로 정렬한다.
  - 오늘 행동 브리핑과 우선 학생 요약을 같은 hero 안에서 묶어 교강사가 먼저 무엇을 할지 바로 읽게 만든다.
  - 학생 큐, 학생 상세, 액션 보드가 같은 디자인 시스템 안에서 이어지도록 CSS 토큰을 정리한다.
- 논의 필요
  - 이번 차수에서 메모 입력을 로컬 상호작용까지만 둘지, 서버 저장까지 연결할지 판단이 필요하다.
- 선택지
  - A. 로컬 상호작용으로 닫고 이후 라운드에서 저장을 붙인다.
  - B. 이번 차수에서 바로 저장 API까지 추가한다.
- 추천
  - A. 현재 round 2는 상단 브리핑과 실제 화면 전환 완성도가 더 중요하다.
- 사용자 방향

## 3차

- 작업내용
  - `pnpm lint`, `pnpm prettier:fix`, `pnpm typecheck`, `pnpm --filter @yeon/web build`를 순서대로 실행한다.
  - 검증 후 현재 landing 단위에 필요한 파일만 stage 해서 커밋, push, PR 생성, 머지까지 진행한다.
- 논의 필요
  - 현재 다중 에이전트 환경에서 같은 브랜치에 이미 존재하는 미추적 기능 파일을 이번 차수에 함께 포함할지 판단이 필요하다.
- 선택지
  - A. 홈 라우트가 의존하는 기능 파일은 함께 포함해 green 상태를 만든다.
  - B. 내 변경만 남기고 홈 라우트를 다시 이전 상태로 되돌린다.
- 추천
  - A. 이미 홈 라우트가 제품형 워크스페이스를 바라보므로, green 기준을 맞추려면 관련 기능 파일을 함께 포함하는 편이 맞다.
- 사용자 방향
