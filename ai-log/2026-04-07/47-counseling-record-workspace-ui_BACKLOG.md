# 상담 기록 워크스페이스 UI 구현 BACKLOG

## 차수 1

### 작업내용

- `apps/web`에 상담 기록 워크스페이스 전용 라우트를 새로 추가한다.
- 왼쪽 기록 네비게이터, 가운데 원문 중심 본문, 오른쪽 AI 상담 도우미로 이루어진 3단 데스크톱 레이아웃을 구현한다.
- 현실적인 한국어 샘플 데이터를 사용해 실제 운영 화면처럼 보이도록 구성한다.

### 논의 필요

- 이 화면을 공개 데모 라우트로 둘지, 로그인 이후 앱 라우트로 편입할지는 후속 제품 흐름과 맞춰 정리가 필요하다.
- 실제 API 연동 전 단계에서 탭, 검색, 채팅 입력은 어디까지 로컬 상호작용으로 표현할지 범위를 정리할 필요가 있다.

### 선택지

- `A. 기존 student-management 라우트 교체`
  현재 운영 화면 자리에 바로 상담 기록 워크스페이스를 배치한다.
- `B. 전용 라우트 신설`
  기존 로그인/운영 흐름과 충돌하지 않도록 독립 라우트와 feature로 구현한다.

### 추천

- `B. 전용 라우트 신설`을 추천한다.
- 이유는 현재 `student-management` 주변 변경이 이미 진행 중이고, 상담 기록 워크스페이스는 별도 landing 단위로 안전하게 분리하는 편이 충돌과 의미 혼선을 줄이기 때문이다.

## 차수 2

### 작업내용

- 첫 구현 후 자체 디자인 리뷰를 수행해 정보 위계, 패널 밸런스, 원문 가독성, AI 패널의 시각적 우선순위를 다시 점검한다.
- 고가치 개선만 반영해 과장 없는 B2B SaaS 톤으로 refinement를 마무리한다.
- 가능한 범위에서 lint, format, typecheck를 실행해 화면 코드의 기본 품질을 확인한다.

### 논의 필요

- 실제 원문 검색, 시간 구간 하이라이트, AI 응답 근거 링크 같은 고급 상호작용은 후속 차수에서 API/상태 설계와 함께 다루는 편이 적절한지 판단이 필요하다.

### 선택지

- `A. 정적 데모 중심`
  현실적인 샘플 데이터와 제한된 상호작용으로 화면 완성도를 우선 확보한다.
- `B. 데모 + 복합 상태 흐름`
  검색, 탭, 채팅, 저장 상태까지 더 깊게 시뮬레이션한다.

### 추천

- `A. 정적 데모 중심`을 추천하되, 검색어 필터링과 채팅 입력 정도만 가볍게 인터랙션으로 제공한다.
- 이유는 이번 목적이 제품 수준의 레이아웃과 워크플로우를 신뢰감 있게 보여주는 것이고, 복합 상태를 과하게 넣으면 오히려 화면 완성도보다 구현 잡음이 커질 수 있기 때문이다.
