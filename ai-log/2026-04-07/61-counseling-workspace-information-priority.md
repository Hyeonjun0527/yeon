# 상담 기록 워크스페이스 정보 우선순위 정의

## 목적

- [60-counseling-workspace-ui-state-map.md](/home/osuma/coding_stuffs/yeon/personal_space/2026-04-07/60-counseling-workspace-ui-state-map.md)을 실제 정보 위계로 번역한다.
- 이후 구현에서는 `무엇을 더 크게 보일지`보다 `무엇을 뒤로 보낼지`를 이 문서를 기준으로 판단한다.
- 현재 문제는 기능 부족이 아니라 `동시에 너무 많이 보이는 것`이므로, 정보 자체를 삭제하지 않고 노출 위계만 다시 잡는다.

## 분류 기준

## 1. 1차 정보

- 사용자가 지금 당장 봐야 다음 행동을 결정할 수 있는 정보
- 한 화면에서 최대 1~3개 묶음만 허용
- 크기, 대비, 위치에서 가장 먼저 읽힌다

## 2. 2차 정보

- 1차 행동을 보조하지만 없어도 다음 행동은 가능하게 만드는 정보
- 본문 근처에 남기되, 강한 카드/색상 강조는 금지

## 3. 보조 정보

- 컨텍스트나 운영 확인용 정보
- 보이더라도 작은 글자, 약한 칩, 접힘 영역, hover, 토글 뒤에 둔다

## 4. 숨김 후보

- 초기 화면에서 노출 가치가 낮고, 오히려 집중을 깨는 정보
- 토글 뒤, 선택 후, 필요 시에만 노출, 또는 완전 제거 대상

## 위계 판단 질문

1. 이 정보가 없으면 사용자가 다음 행동을 못 정하는가
2. 이 정보가 지금 읽히지 않아도 작업이 진행되는가
3. 이 정보는 선택 후에야 의미가 생기는가
4. 이 정보는 운영자에게만 의미 있고 첫 인상에는 방해가 되는가
5. 이 정보는 문장 대신 구조로 전달할 수 있는가

## 글로벌 우선순위 규칙

### 항상 1차여야 하는 것

- 첫 화면의 `파일 업로드`
- 첫 화면의 `브라우저 녹음`
- 선택 후 화면의 `원문 전체`

### 항상 2차 이하여야 하는 것

- 기록 수, 준비 완료 수, 세그먼트 수
- 필터 카운트
- 전사 모델명
- AI 패널의 문맥 설명
- 업로드 제한과 파일 보관 방식 설명

### 선택 후에만 1차로 올라갈 수 있는 것

- 선택 기록 제목
- 현재 전사 상태
- 원문 검색
- AI 질문 입력

## 상태별 위계 매트릭스

## 상태 1. 첫 진입

### 1차 정보

- `파일 업로드`
- `브라우저 녹음`
- 페이지 제목

### 2차 정보

- 한 줄 보조 문구
- `기록을 만들면 원문과 요약이 여기 열립니다` 수준의 짧은 중앙 안내
- 약한 보조 액션 1개

### 보조 정보

- 없음에 가깝게 유지

### 숨김 후보

- 상단 메트릭
- 학생명/학년/상담유형 입력
- 검색창
- 필터칩
- AI 패널
- 긴 제품 소개 문장

### 구현 메모

- 첫 화면은 CTA 두 개가 타이틀보다 더 커도 괜찮지만, 다른 요소가 이 둘보다 강하면 안 된다.
- 중앙 안내는 카드라기보다 여백 위의 한 덩어리 텍스트 감각이면 충분하다.

## 상태 2. 생성 흐름

### 1차 정보

- 선택한 오디오의 존재 여부
- `저장` 액션
- 현재 녹음/업로드 진행 상태

### 2차 정보

- 파일명, 길이, 미리듣기
- `추가 정보` 토글
- 생성 중 짧은 진행 문구

### 보조 정보

- 업로드 용량 제한
- 학생명, 학년/반, 상담 유형, 제목
- 성공/오류 보조 메시지

### 숨김 후보

- AI 패널
- 상세 헤더
- 요약 카드
- 기록 수 요약
- 필터칩

### 구현 메모

- 저장 전 화면에서 가장 강한 정보는 `무엇을 저장하려는지`와 `지금 저장 가능한지`다.
- 학생 메타데이터는 생성 품질을 높이는 정보이지, 생성 시작을 막는 정보가 아니다.

## 상태 3. 목록 탐색

### 1차 정보

- 기록 리스트
- `새 기록 만들기`
- 선택 유도 중앙 안내

### 2차 정보

- 검색창
- 최근 기록 또는 방금 만든 기록 빠른 액션
- 각 기록의 학생명, 제목, 날짜/길이, 현재 상태

### 보조 정보

- 태그
- 미리보기 문장
- 전체 n건, 준비 완료 n건
- 필터 토글

### 숨김 후보

- 필터칩 전체 상시 노출
- AI 패널
- 원문 탭
- 요약/액션 패널
- 상세 메타칩

### 구현 메모

- 목록 탐색 상태에서 가장 중요한 질문은 `어떤 기록을 열까`다.
- 검색은 필요하지만 주인공은 아니고, 필터는 더더욱 주인공이 아니다.

## 상태 4. 기록 선택 후

### 1차 정보

- 선택 기록 제목
- 현재 전사 상태
- 원문 전체
- AI 질문 입력 또는 AI 열기 액션

### 2차 정보

- 짧은 구조화 요약
- 원본 음성 재생
- 원문 검색
- 후속 액션 리스트

### 보조 정보

- 생성일, 길이, 상담 유형
- 세그먼트 수
- 전사 모델명
- 화자 분리 여부
- AI 컨텍스트 요약
- 퀵프롬프트 나머지

### 숨김 후보

- 상세 헤더 긴 설명문
- 메타칩 다발
- 요약 카드 4장 중첩 구조
- AI 상단 설명문

### 구현 메모

- 선택 후 화면의 시선 흐름은 `제목 -> 원문 -> 짧은 요약 -> AI/액션`이어야 한다.
- `원문`보다 먼저 읽히는 메타칩과 설명문은 전부 낮춰야 한다.

## 문구 우선순위 원칙

### 허용되는 문구

- 동사 시작
- 12자 안팎의 짧은 제목
- 1줄짜리 보조 문구
- 결과 중심 문구

### 금지되는 문구

- 제품 소개성 장문
- `기록을 선택해 원문을 읽고, 핵심 요약을 확인한 뒤...` 같은 절차 설명 장문
- `AI 답변 방식`, `선택 가능 기록 0건`처럼 운영 상태를 제목처럼 보이게 하는 문구

## 시각 표현 우선순위 원칙

### 1차 정보 표현

- 크기와 대비를 가장 크게 사용
- 버튼 또는 본문 시작점에 둔다
- 색상 강조를 써도 되는 영역

### 2차 정보 표현

- 본문 주변에 두되 보더 박스로 과장하지 않는다
- 섹션 제목 또는 가벼운 블록 형태

### 보조 정보 표현

- 작은 텍스트
- 약한 칩
- 접힘 영역
- hover, drawer, 토글 뒤

### 숨김 후보 표현

- 기본 DOM에서 제거하거나
- 상태 조건부 렌더링으로 전환한다

## 구현 시 주의

- 상태를 색상만으로 전달하지 않는다. 상태 배지는 텍스트나 아이콘을 함께 둔다.
- 헤딩 위계를 건너뛰지 않는다. `h1 -> h2 -> h3` 흐름을 유지한다.
- 1차 정보가 4개를 넘기기 시작하면 반드시 하나를 2차 이하로 내린다.
- `보여줄 수 있다`와 `지금 보여줘야 한다`를 구분한다.

## 차수 4 입력값

- 차수 4에서는 이 우선순위 문서를 기준으로 실제 디자인 시스템 메모를 작성한다.
- 색상, 간격, 보더 강도, 카드 사용 수, 타이포그래피가 이 위계를 어떻게 지원해야 하는지로 넘어간다.
