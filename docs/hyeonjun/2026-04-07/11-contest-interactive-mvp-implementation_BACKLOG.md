# 11-contest-interactive-mvp-implementation_BACKLOG

## 현재 상태

- 공모전 제출용 문서 세트와 교강사 중심 문제 정의는 이미 정리되어 있다.
- 웹 홈은 현재 `ContestHome` 기반 설명형 MVP로 바뀌어 있어, 제품 메시지와 공용 JSON endpoint는 확인 가능하다.
- `packages/api-contract/src/contest.ts`, `apps/web/src/lib/contest-overview.ts`, `apps/web/src/app/api/v1/contest/overview/route.ts`까지는 연결되어 있지만, 실제 제품 데모처럼 보이는 상호작용은 아직 없다.
- 다음 구현에서 범위를 과하게 키우면 `공모전 제출용 MVP`가 아니라 미완성 LMS처럼 보일 위험이 있다.

## 목표 상태

- 심사자가 첫 화면에서 `오늘 케어 학생`, `위험 이유`, `다음 행동`을 즉시 이해할 수 있는 교강사 중심 데모가 있다.
- 설명형 문서, 앱 로컬 source data, 공용 API 계약, 웹 UI가 같은 구조를 보도록 정렬된다.
- 학생 케어 리스트 -> 학생 상세 케어 카드 -> 피드백 초안 -> 케어 이력 -> 주간 리포트까지 하나의 닫힌 흐름으로 시연 가능하다.
- 모든 차수는 한 번의 프롬프트 또는 짧은 구현 묶음으로 끝낼 수 있는 크기로 유지한다.

## 구현 원칙

- 이번 구현은 `설명형 랜딩`이 아니라 `교강사가 실제로 관리하는 화면`으로 옮겨가는 작업이다.
- AI는 자동 판단보다 `위험 이유 정리`와 `피드백 초안 제안`에 집중시킨다.
- source of truth는 `packages/api-contract`와 `apps/web/src/lib`의 mock/source data에서 먼저 고정한다.
- UI는 차트보다 행동 우선 구조를 택한다.
- 이번 차수에서는 인증, 실제 DB 저장, LLM 실호출보다 `데모 완결성`을 먼저 확보한다.

## Landing 분리 제안

- 1 landing: 공용 contest 도메인 구조와 mock dataset 확장
- 2 landing: 교강사 메인 대시보드 + 오늘 케어 학생 리스트
- 3 landing: 학생 상세 케어 카드 + 피드백 초안 인터랙션
- 4 landing: 케어 이력 + 주간 리포트 + 데모 polish

## 1차

### 작업내용

- `contest overview` 중심 구조를 실제 대시보드용 구조로 확장한다.
- 학생 목록, 위험 신호, 추천 행동, 케어 이력, 주간 리포트에 필요한 mock domain shape를 정의한다.
- `packages/api-contract`와 `apps/web/src/lib`에 같은 구조를 반영한다.

### 논의 필요

- 첫 구현에서 공용 계약을 `overview + dashboard`로 나눌지, `instructor dashboard` 하나로 바로 묶을지

### 선택지

- A. 기존 `overview`를 유지하고 `dashboard` 응답을 별도로 추가
- B. 기존 `overview` 안에 모든 데모 데이터를 몰아서 확장

### 추천

- A. 제출 설명 데이터와 제품 데모 데이터를 분리해야 이후 확장 시 계약 의미가 덜 흐려진다.

### 사용자 방향

## 2차

### 작업내용

- 교강사 메인 대시보드의 정보 구조를 고정한다.
- 첫 화면에 `오늘 케어 학생 수`, `우선순위 3명`, `반복 개념`, `즉시 행동`을 배치한다.
- 현재 `ContestHome`를 유지할지, 대시보드형 홈으로 교체할지 결정한다.

### 논의 필요

- 첫 화면을 공모전 설명형 hero로 유지할지, 바로 교강사 대시보드로 전환할지

### 선택지

- A. 설명형 hero를 상단에 남기고 대시보드를 하단에 추가
- B. 홈 전체를 교강사 대시보드로 전환하고 설명은 보조 섹션으로 축소

### 추천

- B. 심사와 데모 모두 첫 인상은 `제품이 실제로 무엇을 해주는가`가 더 중요하다.

### 사용자 방향

## 3차

### 작업내용

- `오늘 케어 학생 리스트`를 구현한다.
- 학생별로 이름, 위험도, 위험 이유, 최근 변화, 추천 행동을 한 카드에 노출한다.
- 위험 근거는 숫자보다 문장과 태그 중심으로 표현한다.

### 논의 필요

- 위험도를 점수 중심으로 보여줄지, 우선순위 레벨 중심으로 보여줄지

### 선택지

- A. `상/중/하 + 근거 문장`
- B. `점수 + 근거 문장`

### 추천

- A. 공모전 MVP에서는 실측 모델처럼 보이는 숫자보다 해석 가능한 문장이 설득력이 높다.

### 사용자 방향

## 4차

- 최근 학습 신호 타임라인, 위험 해석, 추천 개입 행동, 이전 케어 요약을 한 화면에 묶는다.
- 학생 리스트 선택과 상세 카드 상태를 같은 source data에서 연결한다.

### 논의 필요

- A. 2열 마스터-디테일
- B. 카드 클릭 시 모달/시트

### 추천

- A. 데모 중 학생을 연속 비교하기 쉽고, 교강사 업무 화면처럼 보이기 좋다.

### 사용자 방향

## 5차

### 작업내용

- `1클릭 피드백 초안` 인터랙션을 구현한다.
- 메시지 초안, 면담 제안, 보강 과제 제안 중 하나를 선택하고 수정하는 흐름을 만든다.
- 아직 실제 LLM 호출은 붙이지 않고, mock strategy로 초안 타입만 전환되게 설계한다.

### 논의 필요

- 이번 차수에서 초안 재생성 버튼까지 넣을지, 유형 전환만 제공할지

### 선택지

- A. 초안 유형 선택 + 수정 + 저장
- B. 초안 재생성 인터랙션까지 포함

### 추천

- A. 실제 AI 호출이 없는 상태에서 재생성 UX까지 넣으면 가짜 인터랙션처럼 보일 위험이 있다.

### 사용자 방향

## 6차

### 작업내용

- 학생별 케어 이력 영역을 구현한다.
- 언제 어떤 개입을 했는지, 후속 확인이 필요한지, 다음 액션이 무엇인지 정리한다.
- 피드백 초안 수정 결과가 이력 mock state에 반영되는 로컬 인터랙션을 붙인다.

### 논의 필요

- 이번 차수에서 저장을 서버 route까지 연결할지, 로컬 상태 데모로 마무리할지

### 선택지

- A. 로컬 상태 기반 데모로 우선 마무리
- B. Next route handler까지 연결한 mock write API 추가

### 추천

- A. 공모전 데모에서는 저장 성공 흐름만 안정적으로 보여주는 편이 낫고, 쓰기 API는 다음 landing으로 분리하는 편이 안전하다.

### 논의 필요

- 리포트를 별도 탭/섹션으로 둘지, 홈 하단 요약 카드로 둘지

- B. 별도 리포트 화면

### 추천

- A. 현재 MVP 범위에서는 화면 분기보다 한 페이지 데모 완결성이 더 중요하다.

## 선행 결정 메모

- 기본 사용자 흐름은 `교강사 메인 대시보드 -> 오늘 케어 학생 선택 -> 상세 케어 카드 -> 피드백 초안 수정 -> 케어 이력 확인 -> 주간 리포트 확인`으로 고정한다.
