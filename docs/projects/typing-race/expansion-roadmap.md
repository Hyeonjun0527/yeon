# Typing Race Expansion Roadmap

기준 시점: 2026-04-20

이 문서는 `typing-race` 서비스가 초기 온프레미스 MVP에서 더 큰 운영 형태로 커질 때 어떤 순서로 확장할지 정리한다.
핵심 원칙은 "코드 경계를 먼저 잘라 두고, 나중에 서버 스펙과 배치만 바꾼다"이다.

## 1. 확장 원칙

- 클라우드 과금형 기능을 전제로 설계하지 않는다.
- 운영 환경이 바뀌어도 `apps/web`, `race-server`, `typing-race-engine`, `race-shared` 경계는 유지한다.
- 수평 확장은 실제 병목이 측정된 뒤에만 한다.
- 경기 중 state는 메모리/Redis 중심으로 유지하고, Postgres는 기록 저장에 집중한다.

## 2. 단계별 확장

### 단계 A. Pi 5 단일 장비 MVP

- 대상: 내부 테스트, 소규모 공개 테스트
- 구성: `caddy + web + race-server + redis + postgres`
- 목표:
  - 동시접속 100명 수준에서 UX 깨짐 여부 확인
  - 카운트다운/진행률/결과 정합성 검증

### 단계 B. 강한 단일 서버로 이전

- 대상: 공개 베타 초기
- 구성: 단일 장비 유지, CPU/RAM/SSD만 상향
- 목표:
  - 운영 단순성 유지
  - 단일 서버에서 room 수와 burst 처리 여유 확보

### 단계 C. 경기 서버 분리

- 대상: 동접 증가, 게임 트래픽과 본서비스 트래픽이 서로 간섭하기 시작할 때
- 구성:
  - 장비 1: `web`, reverse proxy
  - 장비 2: `race-server`
  - 선택: Redis/Postgres는 장비 1 또는 별도
- 목표:
  - 경기 서버 CPU를 독립적으로 확보
  - 본서비스 배포와 경기 서버 배포를 분리

### 단계 D. 데이터 계층 분리

- 대상: room 수가 많아지고 재기동/백업/운영 안정성이 중요해질 때
- 구성:
  - Redis 전용 장비
  - Postgres 전용 장비
  - `race-server` 다중 인스턴스
- 목표:
  - 서비스 재시작과 데이터 계층을 분리
  - 유지보수 시간에도 경기 복원력 확보

## 3. 언제 확장할지 판단하는 지표

### Pi 5 단일 장비에서 바로 확장하지 말아야 하는 경우

- 평균 CPU는 낮고, 특정 순간만 일시적으로 튀는 경우
- 메모리 여유가 충분한 경우
- 네트워크가 아니라 프론트 렌더링이 병목인 경우

### 서버 스펙을 올릴 신호

- room tick 지연이 반복적으로 체감되는 경우
- WebSocket send backlog가 늘어나는 경우
- 카운트다운/완주 판정 지연이 누적되는 경우
- DB flush가 경기 루프에 영향을 주기 시작하는 경우

### 서버를 분리할 신호

- 본서비스 배포와 경기 서버 재시작이 서로 방해되는 경우
- `web`와 `race-server`의 CPU 성격이 달라 한 장비에서 튜닝이 어려운 경우
- 저장/로그/배치 작업이 경기 서버 latency를 흔드는 경우

## 4. 장기 기술 선택 유지 기준

### 유지

- `Next.js play shell + Phaser engine`
- `Node.js + Colyseus`
- `race-shared` 프로토콜 패키지

### 필요 시 교체 가능

- `ws` -> `uWebSockets transport`
- 단일 프로세스 -> 다중 프로세스
- 단일 장비 -> 다중 장비
- 온프레미스 -> self-managed VPS/IDC

### 쉽게 교체하면 안 되는 것

- `apps/web` 안에 엔진 코드와 authoritative state를 뒤섞는 것
- 게임 엔진 패키지와 authoritative server 경계를 합치는 것
- hot path를 DB 중심으로 바꾸는 것

## 5. 추후 확장 후보

### 경쟁 기능

- 시즌 컷라인
- 비동기 고스트 매치
- 챌린지 문장
- 결과 리플레이

### 콘텐츠 기능

- 텍스트 카탈로그
- 난이도/언어별 큐
- 카테고리별 시즌

### 운영 기능

- 관리자용 room 모니터링
- 경기 로그 조회
- rate limit / abuse 대응
- 장비별 deploy profile

## 6. 절대 유지할 원칙

- 구조는 먼저 나누고, 비용 최적화는 그 위에서 한다.
- 운영 장비는 나중에 바꿔도, 경계와 프로토콜은 흔들지 않는다.
- 실시간 레이스 품질을 위해 React 셸, 엔진 렌더 루프, authoritative state를 계속 분리한다.
