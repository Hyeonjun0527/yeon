# 상담서비스 API 경로 · 접근정책 정리 BACKLOG

## 작업 목표

- 상담 서비스 클라이언트 요청이 루트 `/api`에 직접 묶이지 않게 만든다.
- `yeon.world/counseling-service` 기준으로 API와 OAuth 진입 경로를 해석하도록 통일한다.
- 서비스 레지스트리에 선언한 접근 정책이 실제 라우팅에도 반영되게 만든다.
- 지금은 단일 배포를 유지하되, 이후 reverse proxy 또는 독립 배포로 승격할 때 재작업을 줄인다.

## 차수 1. 서비스 경로 기반 API helper 도입

### 작업내용

- 앱 경로 helper를 순수 함수와 React context로 분리한다.
- `/home`와 `/counseling-service` 양쪽에서 같은 규약으로 링크와 API 경로를 계산하게 만든다.
- 상담 서비스 관련 클라이언트 fetch, OAuth 시작 링크, 다운로드 링크를 helper 기준으로 치환한다.

### 논의 필요

- API 경로를 전부 hook/context 기반으로만 풀지, 현재 pathname 기반 fallback도 둘지
- 상담 서비스 내부 유틸 파일까지 helper를 끌어올릴지

### 선택지

- 선택지 A: 각 화면에서 `resolveApiHref`를 직접 사용
- 선택지 B: 공통 helper를 두고 화면과 유틸이 함께 사용

### 추천

- 선택지 B
- 상담 서비스 화면이 늘어날수록 경로 규약이 여기저기 흩어지는 것을 막아야 한다.

### 사용자 방향

## 차수 2. 서비스 접근 정책을 proxy에 연결

### 작업내용

- 서비스 레지스트리에서 pathname 기준으로 서비스 엔트리를 찾는 helper를 만든다.
- `auth-required` 서비스 페이지 접근 시 세션 쿠키가 없으면 루트 로그인으로 보낸다.
- API rewrite와 레거시 redirect는 유지하되, 페이지 접근 정책만 추가한다.

### 논의 필요

- proxy에서는 세션 쿠키 존재만 볼지, 실제 세션 유효성까지 검사할지
- 인증이 필요한 planned 서비스가 생길 때 동일 정책을 바로 재사용할지

### 선택지

- 선택지 A: proxy는 쿠키 존재 여부만 보고, 실제 만료 검증은 앱 서버가 담당
- 선택지 B: proxy 단계에서 세션 DB 검증까지 수행

### 추천

- 선택지 A
- Edge 계층에서는 빠른 진입 제어만 맡기고, 세션 무효화 정리는 기존 앱 서버 흐름을 재사용하는 편이 안정적이다.
