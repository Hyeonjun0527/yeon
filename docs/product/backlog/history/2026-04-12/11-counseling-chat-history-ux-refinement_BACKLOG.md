# 11. 상담 AI 채팅 기록 UX 고도화 및 마이그레이션 적용

## 목표

정식 서버 저장이 들어간 AI 채팅을 실제 운영 화면에서 더 읽기 쉽게 다듬고, 가능하면 로컬 환경에 migration까지 적용해 저장 컬럼을 실제 DB에 반영한다.

---

## 차수 1

### 작업내용

- 저장된 채팅 메시지의 `createdAt`을 홈 패널과 상담 워크스페이스 모두에서 유지하도록 타입/매핑을 확장한다.
- 서버 저장 메시지 → 클라이언트 메시지 매핑 시 시간 정보가 유실되지 않게 한다.

### 논의 필요

- 시간 표시를 절대 시각으로 고정할지, 상대 시각과 병행할지 결정 필요.

### 선택지

- A. 절대 시각만 표시
- B. 상대 시각만 표시
- C. 절대 시각을 기본으로 표시

### 추천

- C. 한국어 운영 화면에서는 절대 시각이 더 명확하고, 구현도 단순하다.

## 차수 2

### 작업내용

- `/home`의 `채팅 기록` 탭을 단순 빈 상태/버블 재출력에서 벗어나, 저장된 대화 수/최근 대화 시각/메시지별 시각이 보이는 형태로 다듬는다.
- 현재 채팅 탭에서도 각 메시지 시각을 읽을 수 있게 한다.

### 논의 필요

- "세션별" 표현을 별도 thread 개념으로 나눌지, 현재 record 단위 단일 스레드로 유지할지 결정 필요.

### 선택지

- A. record 단위 단일 히스토리 유지
- B. 같은 record 안에서 새 채팅마다 별도 세션 분리

### 추천

- A. 현재 저장 모델은 record당 단일 대화 스레드이므로, 우선 그 전제를 명확히 보여주는 방향이 맞다.

## 차수 3

### 작업내용

- 상담 기록 워크스페이스(`features/counseling-record-workspace`)의 메시지 버블에도 시간 표시를 넣어 저장 대화와 현재 대화의 일관성을 맞춘다.
- 웰컴/파생 메시지와 저장 메시지가 섞일 때도 저장 메시지가 우선 노출되도록 유지한다.

### 논의 필요

- 웰컴 메시지는 시간 없이 둘지, 현재 시각을 부여할지 결정 필요.

### 선택지

- A. 파생 메시지는 시간 미표시
- B. 파생 메시지에도 현재 시각 부여

### 추천

- A. DB source of truth와 UI 파생 메시지를 구분하는 편이 정합성이 높다.

### 사용자 방향

## 차수 4

### 작업내용

- `pnpm --filter @yeon/web db:migrate`를 실행해 `assistant_messages` 컬럼을 실제 DB에 반영한다.
- 환경 문제로 실패하면 실패 원인과 영향 범위를 즉시 공유한다.

### 논의 필요

- 로컬 `DATABASE_URL`과 DB 접근 가능 여부가 현재 세션에서 보장되는지 확인 필요.

### 선택지

- A. 환경이 되면 즉시 migrate 실행
- B. 환경이 안 되면 SQL 파일까지만 준비

### 추천

- A 우선. 실패하면 B 상태를 명확히 보고한다.
