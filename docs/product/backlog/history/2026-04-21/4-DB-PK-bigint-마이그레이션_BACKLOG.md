# DB PK를 UUID → bigint identity로 마이그레이션 + 외부 노출 난독화

## 배경

- 현재 22개 schema 파일 중 16개가 `uuid("id").primaryKey().defaultRandom()` (UUID v4 랜덤) PK 사용.
- UUID v4는 B-tree 인덱스 페이지 split을 다발시키고 캐시 locality를 깨므로 INSERT/조회 모두 손해.
- URL/API에 UUID를 그대로 노출 (`/spaces/[spaceId]`) → 가독성·길이·SEO 모두 불리.
- 사용자 결정: **내부는 bigint identity, 외부 노출은 난독화 ID로 분리**. UUID v7 절충안 거부.

## 설계 결정 사항

### 결정 1. PK 타입 — **확정**

- `bigint generated always as identity` (PostgreSQL identity column).
- `bigserial` 대신 identity column을 쓰는 이유: SQL 표준, sequence 권한 누수 방지, 명시적.

**Drizzle bigint 모드 정책**: 모든 bigint 컬럼은 `mode: "bigint"`로 통일.
- `mode: "number"`는 2^53 초과 시 silent overflow 위험. 사용 금지.
- TypeScript 측에선 `bigint` 타입으로 다루고, 비교/연산/조인 시 일관 사용.
- JSON 직렬화 시점(외부 노출 차단 layer)에서 bigint은 절대 응답에 담지 않음 — public_id로 변환 후 응답.

### 결정 2. 외부 노출 ID 방식 — **확정**

**확정**: **B + Stripe 스타일 prefix** — `<prefix>_<nanoid 12>` (예: `spc_Vk3xQpZmL_8d`, `mem_8aQrT2bNp_4f`)

**저장**: `public_id text unique not null` (prefix 포함 전체 저장).

**prefix 규약**

| 테이블 | prefix |
|---|---|
| spaces | `spc` |
| members | `mem` |
| member-fields | `mfd` |
| member-tabs | `mtb` |
| member-field-values | `mfv` |
| space-templates | `tpl` |
| space-member-boards | `brd` |
| space-member-board-history | `bhi` |
| public-check-sessions | `pcs` |
| public-check-submissions | `pcb` |
| sheet-integrations | `sht` |
| sheet-integration-member-snapshots | `shs` |
| googledrive-tokens | `gdt` |
| onedrive-tokens | `odt` |
| home-insight-banner-dismissals | `hbd` |
| import-drafts | `idr` |

**이득**
- 로그/디버깅 시 ID만 보고 리소스 즉시 식별
- ID 클래스 오주입 차단 (route 진입점에서 `parseId('spc', raw)` 검증)
- API DX 향상 (Stripe SDK 패턴)
- 사용자 URL 단축 (UUID 36자 → 16자)

**사용자 방향**: 확정 (prefix 사용, nanoid 12자, 위 prefix 표 사용)

### 결정 3. 마이그레이션 전략 — 논의 필요

| 전략 | 장점 | 단점 |
|---|---|---|
| **빅뱅 (한 번에 다 교체)** | 짧은 기간에 끝, 하이브리드 코드 없음 | 다운타임/대규모 코드 변경 동시 발생, 롤백 어려움 |
| **점진 (UUID·bigint 병존 → 점차 전환)** | 무중단, 단계별 롤백 가능 | 개발 기간 길고 임시 컬럼·코드 누적 |

**현재 데이터 규모 가정**: 운영 row 수가 거의 없음 (사용자 확인). 다운타임 허용.

**확정**: **빅뱅** — develop 브랜치에서 도메인 단위 PR 시리즈로 진행.

### 결정 4. 외부 노출 ID 적용 범위 — **확정**

**확정**: **a — 모든 라우트/API에서 `public_id`만 노출**, 내부 bigint은 절대 외부 비공개.

## 차수 계획

### 1차: PoC + 공용 헬퍼 구축

**작업내용**
- `apps/web` (또는 `packages/utils`)에 ID 헬퍼 모듈 추가:
  - `nanoid` 패키지 도입
  - `IdPrefix` const map (위 prefix 표)
  - `generatePublicId(prefix: IdPrefix): string` — `${prefix}_${nanoid(12)}`
  - `parsePublicId(prefix: IdPrefix, raw: string): string` — prefix 검증, 실패 시 `BadRequestError` throw
- 가장 작은 테이블 1개 PoC: **`home-insight-banner-dismissals`** (`hbd` prefix, FK 적음)
  - schema: `id bigint identity` + `public_id text unique` + 기존 user FK는 그대로 유지(아직 users는 v4)
  - 마이그레이션: drop & recreate (PoC 단계, 데이터 거의 없음)
  - service/route 코드를 새 헬퍼로 전환
  - 응답 직렬화: bigint 미노출 검증
- 검증: lint → typecheck → build → 수동 테스트(배너 dismiss flow)

**확정 사항** (모두 사용자 승인)
- nanoid 길이 12자
- 컬럼명 `public_id`
- prefix 패턴 사용 (`<prefix>_<nanoid>`)
- bigint mode = `bigint`

**사용자 방향**: 확정 — 진행

### 2차: 새 PK 컬럼/외부 노출 컬럼 추가 (DDL only)

**작업내용**
- 16개 테이블에 다음 컬럼 추가 (기존 UUID 컬럼 유지):
  - `id_new bigint generated by default as identity` (nullable, 백필 후 not null)
  - `public_id text` (unique, nanoid 백필)
- FK 컬럼들도 `*_id_new bigint` 추가.
- 마이그레이션 SQL로 기존 row의 `id_new` 자동 채움 + `public_id`는 nanoid 생성 후 백필.
- 새 컬럼에 unique index, FK constraint(deferrable) 부여.

**논의 필요**
- 백필 SQL 수행 시 lock 영향 (테이블이 작으면 무시 가능).

**선택지**
- a. 단일 마이그레이션에서 모두 추가
- b. 테이블별 마이그레이션 분리

**추천**: b (롤백 단위 작게).

### 3차: 코드 레이어 전환 (서비스 → API → 라우팅)

**작업내용**
- Drizzle schema에서 `id_new`/`public_id`를 `id`/`publicId`로 alias하고 기존 UUID 컬럼은 `legacyId`로 임시 alias.
- `packages/api-contract` 전체 ID 타입을 string(uuid) → string(public_id) + 내부 처리는 number/bigint로 명시.

## 리스크 정리

1. **파급 범위가 매우 큼** (api-contract, web app, mobile 공용 계약 모두 영향). 동시에 진행 중인 다른 PR과 충돌 가능.
2. **운영 데이터 백필 시 lock 위험** — row 수가 작으면 무시. 운영 row 수 사전 확인 필요.
3. **외부 공개된 URL이 있다면 깨짐** — 현재 공개 체크폼 외에는 내부 사용이라 가정.
4. **bigint 직렬화 함정** — JS `number`는 2^53까지만 안전. JSON으로 직렬화 시 `string`으로 보내거나 `BigInt` 처리 일관성 필요.
