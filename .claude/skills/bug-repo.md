# Bug Repo

프로젝트 코드베이스 전체를 탐색하여 잠재적 버그를 찾아낸다.

## 절대 원칙

- **억지로 채우지 않는다.** 실제로 버그 가능성이 있는 것만 보고한다.
- 최소 0개, 최대 10개. 잠재 버그가 없으면 "잠재 버그 없음"이라고 보고한다.
- 각 항목은 반드시 구체적인 파일 경로와 라인 번호를 포함한다.
- 추측이 아닌, 코드에서 직접 확인한 사실만 근거로 삼는다.
- 이미 알려진 TODO/FIXME는 제외한다. 작성자가 인지하지 못한 문제만 다룬다.
- **동시 작업 인식**: 이 저장소에는 여러 에이전트가 동시에 작업 중일 수 있다. 코드가 미완성처럼 보이거나 임시 구현처럼 보여도 현재 다른 에이전트가 작업 중인 WIP 코드일 수 있다. "이게 버그인가 아니면 진행 중인 작업인가"를 구분하고, 불확실하면 보고 시 "(WIP 가능성 있음)" 주석을 달아 명시한다.

## 탐색 범위

1. `apps/web/src/` — 서버 액션, 라우트 핸들러, 컴포넌트, 훅, 유틸
2. `apps/mobile/` — 존재하면 함께 탐색
3. `packages/` — 공용 패키지
4. 설정 파일 — `next.config.*`, `tsconfig.*`, `tailwind.config.*` 등

## 사고 원칙

탐색 시 아래 사고 방식을 적극 적용한다.

- **논리적 사고**: 코드 흐름을 전제 → 중간 단계 → 결과로 엄밀하게 추론한다. "이럴 것 같다"가 아니라 "이 조건이 성립하면 반드시 이 결과가 나온다"를 증명한다.
- **문제해결적 사고**: 버그가 발생했을 때 어떤 사용자가 어떤 행동을 해야 재현되는지 구체적 시나리오를 만들어 검증한다.
- **복잡한 논리 흐름 사고**: 비동기 체인, 상태 전이, 이벤트 순서 등 여러 단계가 얽힌 흐름을 처음부터 끝까지 머릿속에서 실행해본다.
- **시나리오적 사고**: 정상 경로뿐 아니라 "느린 네트워크", "빠른 연속 클릭", "탭 전환", "페이지 이탈", "빈 데이터", "대용량 데이터" 등 극단 시나리오를 의도적으로 구성한다.
- **보수적 사고**: "이 코드가 맞을 것"이라는 낙관적 전제를 배제하고, 모든 외부 입력과 비동기 타이밍을 적대적으로 가정한다.
- **걱정이 많은 사고**: 사소해 보이는 코드도 "이게 실패하면 무슨 일이 벌어지지?", "이 경로가 두 번 실행되면?", "이 값이 null이면?"을 끊임없이 질문하며 탐색한다.

## 탐색 기준

아래 카테고리를 순서대로 점검한다. 해당 사항이 없으면 건너뛴다.

### P0 — 런타임 크래시 가능성

1. **null/undefined 접근**: 옵셔널 체이닝 없이 중첩 프로퍼티 접근, 배열 인덱스 직접 접근
2. **타입 단언 남용**: `as any`, `as unknown as T`, `!` (non-null assertion)이 실제 null 가능한 값에 사용
3. **에러 삼킴**: catch 블록에서 에러를 무시하거나 빈 catch, 사용자에게 피드백 없는 실패
4. **무한 루프/재귀**: 탈출 조건이 불확실한 반복, useEffect 의존성 배열 문제로 인한 무한 리렌더

### P1 — 데이터 손실 / 상태 오염

5. **레이스 컨디션**: 비동기 호출 간 순서 보장 없음, stale closure, 취소되지 않는 요청
6. **상태 불일치**: 서버/클라이언트 상태 동기화 누락, 낙관적 업데이트 후 롤백 누락
7. **cleanup 누락**: useEffect cleanup 미반환, 이벤트 리스너 미해제, 타이머 미정리
8. **환경변수 미검증**: 런타임에 undefined가 될 수 있는 환경변수를 검증 없이 사용

### P2 — 엣지 케이스 / 조용한 오작동

9. **경계값 미처리**: 빈 배열, 빈 문자열, 0, 음수 등 경계값에서 예상과 다른 동작
10. **의존성 배열 오류**: useEffect/useMemo/useCallback의 deps 누락 또는 과잉
11. **조건 분기 구멍**: if/else, switch에서 처리되지 않는 케이스
12. **타입과 런타임 불일치**: TypeScript 타입은 맞지만 런타임 데이터가 다를 수 있는 지점 (외부 API 응답 등)

## 실행 절차

1. 탐색 범위 내 파일 목록을 수집한다.
2. 파일별로 위 탐색 기준을 적용하며 읽는다.
3. 발견된 잠재 버그를 severity로 분류한다.
4. 각 항목에 재현 시나리오 또는 트리거 조건을 명시한다.

## 출력 형식

```
## 잠재 버그 탐색 결과: N개 발견

### P0 — 런타임 크래시 가능성
- **[제목]** `파일경로:라인`
  - 증상: 어떤 상황에서 어떤 문제가 발생하는지
  - 트리거: 이 버그가 발현되는 조건
  - 권장 수정: 간단한 수정 방향

### P1 — 데이터 손실 / 상태 오염
- **[제목]** `파일경로:라인`
  - 증상: ...
  - 트리거: ...
  - 권장 수정: ...

### P2 — 엣지 케이스 / 조용한 오작동
- **[제목]** `파일경로:라인`
  - 증상: ...
  - 트리거: ...
  - 권장 수정: ...

(해당 severity에 항목이 없으면 해당 섹션 생략)
```

잠재 버그가 하나도 없으면:

```
## 잠재 버그 탐색 결과: 0개

현재 코드베이스에서 탐색 기준에 해당하는 잠재 버그를 발견하지 못했습니다.
```
