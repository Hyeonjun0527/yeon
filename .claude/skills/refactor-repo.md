# Refactor Repo

현재 프로젝트의 코드베이스를 전체적으로 점검하고, 실제로 리팩토링이 필요한 지점을 도출한다.

## 절대 원칙

- **억지로 10개를 채우지 않는다.** 실제로 리팩토링 가치가 있는 것만 보고한다.
- 최소 0개, 최대 10개. 리팩토링 대상이 없으면 "리팩토링 대상 없음"이라고 보고한다.
- 각 포인트는 반드시 구체적인 파일 경로와 라인 번호를 포함한다.
- "이러면 더 좋을 것 같다" 수준의 취향 차이는 포함하지 않는다.
- 리팩토링의 이유와 기대 효과가 명확해야 한다.

## 점검 기준 (CLAUDE.md / CLAUDE.local.md 기반)

아래 기준을 순서대로 점검하되, 해당 사항이 없으면 건너뛴다.

### 구조적 리팩토링 (높은 가치)

1. **SRP 위반**: 하나의 파일/함수/컴포넌트가 여러 이유로 변경되는 구조
2. **source of truth 분산**: 같은 의미의 raw 문자열이 여러 분기에 흩어져 있어 상수 객체로 승격해야 하는 경우
3. **깊은 조건 분기**: 조기 반환, 매핑 테이블, 보조 함수 분리로 단순화 가능한 중첩 if
4. **workspace 경계 정리**: apps/packages 간 잘못된 의존 방향, 책임 혼재

### 코드 품질 리팩토링 (중간 가치)

5. **중복 로직**: 3곳 이상에서 동일 패턴이 반복되어 추상화 가치가 있는 경우 (2곳은 허용)
6. **stale derived state**: 원본이 바뀌었는데 파생값이 갱신되지 않는 구조
7. **불필요한 복잡도**: 과잉 추상화, 불필요한 wrapper, 사용되지 않는 제네릭
8. **비동기 구조 개선**: 취소 패턴 부재, 불필요한 waterfall, 병렬화 가능한 순차 호출

### 정리 리팩토링 (낮은 가치)

9. **dead code**: import되지 않는 export, 도달 불가능한 분기, 사용되지 않는 변수/함수
10. **네이밍/구조 불일치**: 파일명과 export명 불일치, 폴더 구조와 실제 책임 불일치

## 실행 절차

1. `apps/web/src/` 아래 주요 feature, component, server 코드를 탐색한다.
2. `packages/` 아래 코드가 있다면 함께 점검한다.
3. 위 점검 기준에 해당하는 실제 리팩토링 대상만 수집한다.
4. 각 포인트를 가치 수준 (높음/중간/낮음)으로 분류한다.

## 출력 형식

```
## 리팩토링 대상: N개 발견

### 높은 가치
- **[제목]** `파일경로:라인` — 현재 문제, 리팩토링 방향, 기대 효과.

### 중간 가치
- **[제목]** `파일경로:라인` — 현재 문제, 리팩토링 방향, 기대 효과.

### 낮은 가치
- **[제목]** `파일경로:라인` — 현재 문제, 리팩토링 방향, 기대 효과.

(해당 가치 수준에 포인트가 없으면 해당 섹션 생략)
```

해당 사항이 하나도 없으면:

```
## 리팩토링 대상: 0개

현재 코드베이스에서 점검 기준에 해당하는 리팩토링 대상을 발견하지 못했습니다.
```
