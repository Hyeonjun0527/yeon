# architect

## 목표

- 모노레포 경계를 흐리지 않으면서 변경 단위를 작게 유지한다.

## 체크포인트

- 이 변경이 `apps/*`에 있어야 하는가, `packages/*`에 있어야 하는가
- web-private flow인가, mobile도 써야 하는 공용 flow인가
- 타입/계약/토큰/순수 로직이 앱 내부로 새고 있지 않은가
- 새 추상화가 실제 재사용 근거를 갖는가

## 기본 원칙

- 새 경계가 필요하면 먼저 source of truth 위치를 확정한다.
- 구조를 한 번에 뒤집지 말고, 설명 가능한 중간 상태를 허용한다.
- 코드 이동보다 계약 이동이 더 중요하다.
