## 차수 1

### 작업내용

- 출석/과제 공개 체크인 흐름에서 리뷰 지적사항을 `즉시 수정 가능 항목`과 `정책 확정 후 수정 항목`으로 다시 분류한다.
- 다음 값을 source of truth로 명시한다.
  - 세션 계약: `checkMode`, `enabledMethods`
  - 공개 제출 계약: `assignmentStatus`, `assignmentLink`, `method`
  - 운영자 보드 원본: 서버의 `student-board`
  - 운영자 입력 중 상태: 클라이언트 `drafts`
- 각 항목별로 “무엇을 바꾸면 기능 의미가 바뀌는지”를 명시해 unsafe patch를 금지한다.

### 논의 필요

- 없음

### 선택지

- 리뷰 코멘트를 순서 없이 바로 코드에 반영한다.
- 위험도가 높은 항목을 정책/상태 전이 기준으로 나눠 차수별로 처리한다.

### 추천

- 위험도가 높은 항목을 정책/상태 전이 기준으로 나눠 차수별로 처리한다. 지금 문제는 버그 수정 자체보다 “수정하다가 기능 의미가 바뀌는 것”이 더 큰 리스크다.

### 사용자 방향

## 차수 2

### 작업내용

- 비교적 안전한 항목만 먼저 수정한다.
  - `location only` 세션에서 기본 `method`가 허용된 값으로만 유지되게 수정

- 의미 보존이 확실한 입력 검증/기본값 동기화부터 먼저 고친다.

### 추천

- 의미 보존이 확실한 입력 검증/기본값 동기화부터 먼저 고친다. 이 단계는 회귀 위험이 낮고, 실제 사용자 실패를 즉시 줄일 수 있다.

## 차수 3

### 작업내용

- `assignmentStatus: "unknown"` 처리 정책을 코드보다 먼저 고정한다.
- 아래 세 가지를 명시적으로 분리한다.
  - 과제 상태를 명시 선택한 제출
  - 과제 링크만 제출한 제출
  - 과제 정보를 실질적으로 제출하지 않은 제출
- 그 정책에 따라 서버 저장 규칙을 분리한다.
  - `spaceMemberBoards.assignmentMarkedAt/Source`를 언제 찍을지
  - `publicCheckSubmissions.assignmentStatus/assignmentLink`를 언제 남길지
  - 결과 메시지를 언제 “과제 체크 완료”로 볼지
- 정책 고정 후 UI도 맞춘다.
  - 필요한 경우 `과제 상태 선택`을 필수화
  - 혹은 링크-only 제출을 별도 안내 문구로 분리

### 논의 필요

- `assignmentLink`만 있어도 과제 제출로 볼지 여부

### 선택지

- `done`/`not_done` 선택 시에만 과제 체크로 인정한다.
- `assignmentLink`만 있어도 과제 제출로 인정한다.
- `assignmentLink`는 부가 메모일 뿐, 상태 선택이 없으면 과제 제출로 인정하지 않는다.

### 추천

- 상태 선택(`done`/`not_done`)이 있을 때만 과제 체크로 인정하고, 링크-only는 링크 저장으로만 처리한다. 과제 완료/미완료와 링크 첨부는 의미가 다르므로 섞으면 다시 해석 혼선이 생긴다.

### 사용자 방향

## 차수 4

### 작업내용

- 운영자 보드의 `drafts` 동기화 정책을 재설계한다.
- `drafts`를 서버 원본의 복사본으로 보지 말고, “미저장 사용자 입력”으로 한정한다.
- 상태를 아래로 분리한다.
  - 서버 원본 row
  - dirty 여부
  - 마지막 hydrate 기준값
- 동기화 규칙을 고정한다.
  - dirty가 아닌 row만 refetch 시 서버값으로 갱신
  - dirty row는 저장 또는 명시 취소 전까지 유지
  - 저장 성공 시 dirty 해제 후 서버 응답으로 동기화
- 세션 생성/열기/닫기 invalidate가 있어도 draft 유실이 없도록 보장한다.

### 논의 필요

- 없음

### 선택지

- 서버 refetch 시 draft 전체 동기화
- dirty row만 보호
- 서버 원본과 draft를 별도 store로 분리

### 추천

- dirty row만 보호하는 방식으로 시작한다. 구조를 과하게 키우지 않으면서도 지금 문제인 draft 유실과 stale state를 동시에 줄일 수 있다.

### 사용자 방향

## 차수 5

### 작업내용

- `isComposerOpen`의 초기 자동 결정과 사용자 수동 토글을 분리한다.
- 로딩 완료 전에는 미결정 상태를 유지하고, 최초 성공 응답에서만 기본 open/close를 계산한다.
- 사용자가 한 번이라도 토글하면 이후 refetch로는 UI를 다시 강제하지 않는다.

### 논의 필요

- 없음

### 선택지

- `boardSessions.length` 변화마다 effect로 강제 동기화
- 최초 hydrate에서만 기본값 결정 후 사용자 제어로 넘김

### 추천

- 최초 hydrate에서만 기본값 결정 후 사용자 제어로 넘긴다. 이게 “초기 자동 편의”와 “사용자 의도 보존”을 동시에 만족한다.

### 사용자 방향

## 차수 6

### 작업내용

- 회귀 테스트를 보강한다.
  - `enabledMethods=["location"]` 기본 제출 성공/실패
  - `latitude=0`, `longitude=0` 세션 생성
  - `assignment_only + unknown`
  - `assignment_only + link only`
  - `attendance_only + crafted assignment payload`
  - dirty draft 유지 / 저장 후 동기화
  - composer 초기 자동 결정 vs 사용자 수동 토글 유지
- 필요하면 컴포넌트 테스트와 서비스 테스트를 분리한다.

### 논의 필요

- 없음

### 선택지

- 서비스 테스트만 추가
- 서비스 테스트 + UI 상태 테스트를 함께 추가

### 추천

- 서비스 테스트 + UI 상태 테스트를 함께 추가한다. 이번 이슈는 서버 계약과 클라이언트 상태가 동시에 얽혀 있어 한쪽 테스트만으로는 재발을 막기 어렵다.
