# 홈 워크스페이스 UI 정책 분리 BACKLOG

## 1차

### 작업내용

- 상담기록 홈과 학생관리 화면에서 `목록 접기` 버튼 노출 여부를 명시적 정책으로 분리한다.
- `Gnav`가 빈 상태나 DOM 구조를 직접 추론하지 않고, 각 화면이 계산한 `canToggleSidebar`를 받도록 정리한다.
- `스페이스 없음`, `상담기록 없음`처럼 실제 접을 대상이 없는 상태에서는 목록 접기 버튼을 숨긴다.

### 논의 필요

- 접기 가능 여부를 기존 레이아웃 컨텍스트에 둘지, 별도 `ui policy` 컨텍스트로 독립시킬지

### 선택지

- A. 기존 `HomeSidebarLayoutContext`에 `canToggleSidebar` 정책을 추가한다.
- B. 별도 `HomeUiPolicyContext`를 새로 만들고 `Gnav`, `TopNav`, 각 화면이 공유한다.

### 추천

- A. 이번 차수는 목록 접기 버튼만 범위이므로 기존 레이아웃 컨텍스트를 확장해도 충분하다. 추후 튜토리얼/첫 화면 정책까지 묶일 때 별도 컨텍스트 승격 여부를 다시 판단한다.

### 사용자 방향

- `1차`만 구현

## 2차

### 작업내용

- 튜토리얼 버튼 노출과 튜토리얼 step 모드를 `empty/full/disabled` 같은 명시적 정책으로 분리한다.
- DOM `data-tutorial` 존재 여부를 직접 source of truth로 쓰는 구조를 줄인다.
- 학생관리/출석보드의 빈 상태 튜토리얼 여부를 화면 정책에서 결정하도록 정리한다.

### 논의 필요

- 빈 상태 튜토리얼을 유지할지, 특정 섹션에서는 완전히 끌지

### 선택지

- A. 빈 상태 튜토리얼 유지, 정책으로만 이동
- B. 빈 상태 튜토리얼 제거, 실사용 상태에서만 노출

### 추천

- A. 현재 안내 가치가 있으므로 기능은 유지하되 DOM 추론 대신 정책 기반으로 전환하는 편이 안전하다.

### 사용자 방향

## 3차

### 작업내용

- 상담기록 홈과 학생관리의 첫 화면 전환(`space-gate`, `empty`, `workspace`)을 명시적 UI 정책으로 분리한다.
- `showStudentShell`, `records.viewState.kind === "empty"` 같은 화면 내부 분기를 정책 객체로 승격한다.
- 이후 `Gnav`, `TopNav`, 튜토리얼, 첫 화면이 같은 정책을 읽도록 경계를 정리한다.

### 논의 필요

- 정책 계산 함수를 각 섹션 로컬 `_lib`에 둘지, 홈 공용 `_lib`로 합칠지

### 선택지

- A. 섹션별 로컬 정책 함수로 유지
- B. 홈 공용 정책 모듈에서 통합 관리

### 추천

- A. 상담기록과 학생관리의 상태 모델이 다르므로 섹션별 로컬 정책 함수가 더 명확하다.
