# chat-service 모바일 MVP 출시기획

| 버전 | 날짜 | 변경 내용 |
| --- | --- | --- |
| v0.1 | 2026-04-23 | 모바일 전용 `chat-service` 1차 기획/작업계획 초안 작성 |

대상 독자: 기획자, 디자이너, 모바일 개발자, 운영 담당자

---

## 1. 기획 배경

- 새 서비스는 `/chat-service`를 기준 이름으로 잡고, Google Play와 iOS App Store 동시 출시를 목표로 한다.
- 제품은 웹보다 모바일에서만 먼저 완성한다.
- 사용자가 제공한 예시 화면처럼 하단 5탭과 고밀도 리스트 UI가 기본 UX 방향이다.
- 다만 예시 화면의 "익명 채팅앱" 톤을 그대로 가져가면 앱스토어 심사 리스크가 크다.
- 그래서 1차 제품 정의는 `랜덤 익명 채팅`이 아니라 `공개 대화 + 질문/투표 + 친구 기반 DM`으로 잡는다.
- deep-interview 결과, 제품의 두 축은 `공개 커뮤니티`와 `친구 기반 DM`이며 둘을 동등하게 본다.
- deep-interview 결과, v1은 전 기능 로그인 전제이고 인증은 소셜로그인이 아닌 간단한 전화번호 인증을 사용한다.
- deep-interview 결과, `chat-service`는 `yeon` 공용 로그인 허브에 붙지 않는 서비스 전용 로그인 경험을 가져야 하지만 backend는 모든 서비스 공용으로 하나를 유지한다.
- deep-interview 결과, DM은 상대 프로필에서 100원 결제로 즉시 오픈된다. 수신 거부는 후속 기능이다.

## 2. 서비스 한 줄 정의

`chat-service`는 공개 대화가 쌓이는 커뮤니티 축과 친구 기반 1:1 대화 축을 동등하게 가진 모바일 소셜 채팅 서비스다. 인증 진입 경험과 세션 namespace는 `yeon` 공용 로그인 허브와 분리하되, backend는 모든 서비스 공용으로 하나를 유지한다.

## 3. 핵심 사용자

| 사용자 | 목적 | 성공 경험 |
| --- | --- | --- |
| 신규 로그인 사용자 | 앱 분위기와 대화 밀도를 빠르게 확인 | 피드/에스크를 보고 서비스 성격을 즉시 이해 |
| 가벼운 대화 사용자 | 짧은 글, 멘션, 댓글로 존재감 표현 | 부담 없이 글을 올리고 반응을 받음 |
| 질문 사용자 | 의견을 묻거나 투표를 올림 | 짧은 시간 안에 답변/투표를 얻음 |
| 관계 형성 사용자 | 자주 보이는 사람과 친구를 맺고 DM 이동 | 공개 대화에서 private chat으로 자연스럽게 이동 |
| 운영 담당자 | 문제 유저/콘텐츠 제어 | 신고 접수, 차단, 숨김, 정지 처리를 빠르게 수행 |

| 5. 프로필설정 | 계정/설정/안전 | 프로필 수정, 알림, 차단 목록, 계정 삭제 |

## 5. 예시 화면에서 가져갈 것과 버릴 것

### 가져갈 것

- 흰 배경 중심의 단순한 모바일 리스트 UI
- 상단 타이틀 + 우측 설정/필터 아이콘
- 닉네임, 지역, 나이, 최근 상태가 한 줄 안에서 빠르게 읽히는 구성
- 하단 5탭 고정 내비게이션

### 버릴 것

- 무차별 익명 채팅으로 오해될 수 있는 제품 포지셔닝
- 포인트 충전/광고를 MVP 핵심 화면으로 전면 배치하는 구조
- 신고/차단/운영 흐름 없이 대화 기능만 먼저 여는 방식

## 6. 출시 전제

- Apple App Review Guidelines 1.2 / 4.8 / 5.1.1과 Google Play user-generated content policy를 출시 전제 조건으로 본다.
- 따라서 MVP 범위에 아래가 반드시 포함된다.
  - 신고
  - 차단
  - 운영 연락처
  - 커뮤니티 가이드라인
  - 인앱 계정 삭제
- v1은 전화번호 인증만 가정하므로 `Sign in with Apple`은 현재 MVP 범위 밖으로 두되, 추후 제3자 로그인 도입 시 Apple 4.8을 재검토한다.
- 랜덤 익명 매칭, 무제한 DM, 운영부재형 오픈채팅은 1차 범위에서 제외한다.
- 공용 로그인 화면과 공용 세션 namespace를 재사용하지 않으므로, 약관/개인정보처리방침/운영 연락처도 `chat-service` 단위로 준비한다.

## 7. 추천 제품 방향

### 핵심 방향

- Feed와 Ask에서 공개적으로 관계가 먼저 생긴다.
- Friends는 관계를 승인/정리하는 경계다.
- Chat은 상대 프로필에서 100원 결제로 오픈되는 1:1 DM 구조로 시작한다.
- Profile은 수익화보다 계정/설정/안전 중심으로 간다.
- 모든 주요 흐름은 로그인 사용자 기준으로 설계한다.
- backend는 공용으로 유지하고, 인증 진입 경험과 세션 namespace는 `chat-service` 전용 경계에서 처리한다.

### 핵심 규칙

- 읽기와 쓰기 모두 1차는 로그인 사용자만 가능하다.
- 로그인은 간단한 전화번호 인증으로 시작한다.
- 백엔드는 모든 서비스 공용으로 하나를 유지한다.
- `yeon` 공용 로그인 화면과 공용 세션 namespace를 재사용하지 않는다.
- DM은 상대 프로필에서 100원 결제로 즉시 열린다.
- 상대방 수신거부 설정은 후속 기능이다.
- 피드/에스크/채팅/프로필 어디서든 신고와 차단에 닿아야 한다.
- 1차는 텍스트 중심으로 제한한다.

## 1차 — 제품 정의 / 정책 / IA 확정 (본 문서)

### 작업내용

- `chat-service`의 1차 제품 정의를 확정한다.
- 앱스토어 출시를 막는 리스크를 제품 정책에 먼저 반영한다.
- 탭 구조, 핵심 엔터티, 공개 영역과 DM 영역의 경계를 정한다.
- 코드 수정 없이 기획/작업계획만 작성한다.

### 논의 필요

1. 제품을 어디까지 익명으로 가져갈지
2. 100원 DM 오픈 정책의 abuse 방지를 어디까지 같이 넣을지
3. 1차에서 포인트/광고를 넣을지
4. 전화번호 인증의 구현 방식과 abuse 방지 수준
5. 단일 backend 안에서 auth를 어느 수준까지 서비스 전용으로 분리할지

### 선택지

**(a) 제품 포지셔닝**
- A1. 랜덤 익명 채팅 중심
- A2. 닉네임 기반 공개 커뮤니티 + 친구 기반 DM
- A3. 질문/투표 앱 + DM은 후순위

**(b) 로그인 경계**
- B1. 읽기/쓰기 모두 로그인 필수
- B2. 읽기는 일부 허용, 쓰기/친구/채팅은 로그인 필수
- B3. 게스트도 대부분 기능 허용

**(c) DM 정책**
- C1. 상대 프로필에서 100원 결제 후 즉시 오픈
- C2. 친구 수락 후 DM 가능
- C3. 채팅 요청 승인 후 DM 가능, 친구와 동일하게 취급하지 않음

**(d) 수익화 시점**
- D1. 1차부터 포인트/충전/광고 포함
- D2. 포인트 지갑만 노출, 결제는 추후
- D3. 1차는 수익화 제외, 설정/안전 중심

### 추천

- (a) **A2**
- (b) **B1**
- (c) **C1**
- (d) **D3**

### 사용자 방향

- 공개 커뮤니티와 친구 기반 DM을 동등하게 가져간다.
- v1은 전체 로그인 필수다.
- 로그인은 소셜로그인이 아니라 간단한 전화번호 인증으로 간다.
- `chat-service`는 공용 로그인 허브 UX에 붙지 않는다.
- backend는 모든 서비스 공용으로 하나를 유지한다.
- DM은 상대 프로필에서 100원 결제로 즉시 오픈한다.
- 수신거부는 매우 나중의 후속 기능이다.

## 2차 — 앱 셸 / 서비스 전용 전화번호 인증 / 프로필 최소 모델

### 작업내용

- `apps/mobile`에 Expo Router 탭 셸을 만든다.
- `피드 / 에스크 / 친구 / 대화 / 프로필설정` 5탭 route를 세운다.
- 서비스 전용 전화번호 인증, 세션 유지, 프로필 최소 모델을 설계한다.
- `packages/api-contract` 기준의 mobile API 계약 초안을 만든다.
- 단일 backend 안에서 `yeon` 공용 로그인과 다른 auth flow/session namespace 경계를 문서화한다.

### 논의 필요

1. 전화번호 인증 구현 방식
2. 프로필 필수값 범위
3. 지역/나이 노출 방식
4. 닉네임 중복 정책
5. 서비스 전용 auth flow와 session namespace 경계

### 선택지

**(a) 전화번호 인증 구현**
- A1. Firebase Phone Auth
- A2. 국내 SMS 벤더 + 자체 OTP 서버
- A3. 외부 auth SaaS 사용

**(e) auth 독립 수준**
- E1. 계정 DB, 세션, 토큰, 약관 동의까지 전부 독립
- E2. DB/인프라는 공유하지만 auth flow와 session namespace만 독립
- E3. 공용 계정 테이블 재사용 + 앱 UX만 분리

**(b) 프로필 필수값**
- B1. 닉네임만
- B2. 닉네임 + 연령대 + 지역
- B3. 닉네임 + 연령대 + 지역 + 성별

**(c) 나이 표현**
- C1. 정확한 출생연도
- C2. `19세`, `20대 초반` 같은 표시용 연령값
- C3. 공개 비노출

**(d) 지역 표현**
- D1. 시/도만 공개
- D2. 시/군/구까지 공개
- D3. 비공개 기본 + 노출 선택

### 추천

- (a) **A1**
- (b) **B2**
- (c) **C2**
- (d) **D1**
- (e) **E2**

### 사용자 방향

- 로그인은 전화번호 인증으로 확정
- 백엔드는 하나로 두고, 공용 로그인 허브에 붙지 않는 전용 auth flow/session namespace로 간다

## 3차 — 피드 / 에스크 MVP

### 작업내용

- Feed 탭에 짧은 공개 글 목록과 작성 플로우를 넣는다.
- Ask 탭에 질문글과 투표글 작성 플로우를 넣는다.
- 피드와 에스크 모두 신고/차단 액션을 포함한다.
- 예시 화면처럼 고밀도 리스트형 레이아웃을 적용한다.
- 로그인한 사용자 전용 홈 표면으로 설계한다.

### 논의 필요

1. 피드 글과 에스크 글의 차이를 어디까지 나눌지
2. 피드 답글을 1차에 넣을지
3. 투표 선택지 개수와 공개 시점
4. 글 길이 제한

### 선택지

**(a) Feed 구조**
- A1. 글만 있고 댓글 없음
- A2. 글 + 1레벨 답글
- A3. 글 + 다단계 스레드

**(b) Ask 구조**
- B1. 질문글만
- B2. 질문글 + 투표글
- B3. 질문글 + 투표글 + 익명답변

**(c) 투표 결과 공개**
- C1. 투표 전에도 결과 보임
- C2. 투표 후 결과 보임
- C3. 작성자만 결과 확인

**(d) 글 형식**
- D1. 텍스트만
- D2. 텍스트 + 이미지 1장
- D3. 텍스트 + 이미지/링크/파일

## 4차 — 친구 / 대화 MVP

### 작업내용

- Friends 탭에 친구 목록, 친구 요청, 추천 사용자 노출을 만든다.
- Chat 탭에 채팅방 목록과 1:1 텍스트 채팅을 넣는다.
- 차단 관계와 친구 관계가 DM 가능 여부에 반영되도록 한다.
- 읽지 않음, 마지막 메시지, 최근 접속 같은 핵심 상태만 우선 노출한다.
- 전화번호 기반 계정 식별과 친구 요청/차단 정책이 관계 모델에 반영되도록 한다.
- 프로필 상세에서 100원 결제로 채팅 오픈하는 플로우를 넣는다.

### 논의 필요

1. 100원 결제 DM 오픈 뒤 친구 관계를 자동 생성할지
2. 채팅 오픈 취소/환불 정책을 둘지
3. 읽음 상태 표시 범위
4. 첨부파일/이미지 전송 시점

### 선택지

**(a) 관계 모델**
- A1. 친구 요청만 존재
- A2. 팔로우 + 친구 요청 혼합
- A3. 친구 개념 없이 채팅 요청만 존재

**(b) DM 오픈 조건**
- B1. 상대 프로필에서 100원 결제 후 즉시 가능
- B2. 친구 수락 후 가능
- B3. 채팅 요청 승인 후 가능

**(c) 채팅 형식**
- C1. 1:1 텍스트만
- C2. 1:1 텍스트 + 이미지
- C3. 그룹채팅 포함

**(d) 대화 정렬**
- D1. 최근 메시지 순
- D2. 고정 친구 우선
- D3. 읽지 않음 우선

## 5차 — 프로필설정 / 안전 / 운영도구

### 작업내용

- Profile 탭을 계정/설정/안전의 진입점으로 만든다.
- 프로필 수정, 알림 설정, 차단 목록, 신고 접수 상태, 커뮤니티 가이드라인, 운영 문의, 계정 삭제를 넣는다.

## 구현 구조 추천

### `apps/mobile`

- `app/(tabs)`는 route shell만 담당
- 실제 화면 조합과 상태는 `src/features/chat-service/*`
- API 호출과 세션/재시도는 `src/services/chat-service/*`

### `packages/*`

- `packages/api-contract`: feed, ask, friend, chat, profile, moderation DTO / schema
- `packages/api-client`: typed mobile client
- `packages/domain`: 차단 판정, friend relation 해석, poll rules 같은 순수 로직
- `packages/design-tokens`: cross-platform color, spacing, radius, icon size

### 금지

- `apps/mobile`에서 `apps/web/src/server` 직접 import
- 앱 내부 DTO를 shared package에 무분별하게 밀어 넣기
- 네트워크 로직을 탭 화면 컴포넌트 안에 직접 퍼뜨리기

## 출시 차단 리스크

1. 랜덤 익명 채팅처럼 보이는 포지셔닝은 Apple 심사 리스크가 높다.
2. 신고/차단/계정삭제가 빠지면 Apple/Google 모두 출시 리스크가 높다.
3. 수신자 동의 없는 100원 DM 오픈은 harassment/abuse 리스크가 크다.
4. 전화번호 인증을 너무 단순하게만 만들면 abuse와 다계정 대응이 약해진다.
5. 단일 backend 안에서 auth flow/session namespace 경계를 흐리면 추후 `yeon` 공용 로그인과 충돌하거나 데이터 경계가 모호해질 수 있다.
6. 포인트/광고/충전을 먼저 넣으면 제품 핵심 가치보다 잡음이 커진다.
7. 운영 admin 없이 공개 출시하면 초반 abuse 대응이 불가능하다.

## 이번 문서의 기본 추천 결론

- 제품은 `공개 대화 + 질문/투표 + 친구 기반 DM`으로 간다.
- 공개 커뮤니티 축과 친구 기반 DM 축을 동등하게 본다.
- 1차는 텍스트 중심, 1:1 DM 중심, 운영 가능성 중심으로 자른다.
- Feed/Ask가 먼저 열리고 Chat은 프로필 기반 100원 결제 오픈 구조로 연결한다.
- Profile은 안전과 설정 중심으로 시작하고, 수익화는 후속으로 미룬다.
- v1은 전체 로그인 필수이며 인증은 간단한 전화번호 인증으로 간다.
- `chat-service`는 백엔드는 하나로 두고, `yeon` 공용 로그인 허브와 분리된 서비스 전용 auth flow/session namespace로 간다.
- 구현 단계에 들어가면 `ralph-strict`의 story-level code review 훅을 붙여 진행한다.
