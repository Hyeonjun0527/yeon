## 1차

### 작업내용

- `/mockdata/app` 데모보기를 실제 제품 구조 기준으로 재설계한다.
- 현재 standalone mock UI를 버리고, 실제 페이지 계층/패널 구조를 반영하는 seeded demo shell로 전환한다.
- 상담 기록 워크스페이스, 수강생 관리, 출석보드, 가져오기 흐름을 실제 라우트 구조와 동일한 정보 위계로 재배치한다.
- 실제 컴포넌트 재사용 범위와 mock 전용 adapter 경계를 분리한다.

### 논의 필요

- 데모가 실제 컴포넌트를 직접 재사용할지, 아니면 실제 구조만 닮은 demo shell을 둘지 결정이 필요하다.
- `/mockdata/app` 하나에서 탭으로 다 보여줄지, `/mockdata/home`, `/mockdata/student-management`처럼 실제 라우트와 유사하게 나눌지 결정이 필요하다.
- mock 데이터 주입 방식을 query/provider 기반으로 할지, mock 전용 fetch layer로 할지 정해야 한다.

### 선택지

1. 실제 컴포넌트 최대 재사용 + mock data provider 주입
   - 장점: 실제 페이지와 시각/상태 구조가 가장 잘 맞는다.
   - 단점: 기존 실제 컴포넌트가 mock 주입을 받을 수 있도록 adapter 작업이 필요하다.
2. mock 전용 화면을 유지하되 실제 구조를 수동으로 다시 작성
   - 장점: 빠르게 화면만 비슷하게 만들 수 있다.
   - 단점: 다시 금방 실제 제품과 어긋난다.
3. 실제 라우트 복제 + mock route 전용 wrapper 추가
   - 장점: 라우트 경험은 맞출 수 있다.
   - 단점: wrapper가 얇지 않으면 중복 구조가 커진다.

### 추천

- 선택지 1을 기본으로 간다.
- 목표는 “데모용 가짜 화면”이 아니라 “실제 제품 구조에 seeded data를 주입한 읽기 중심 데모”다.
- 단일 `/mockdata/app` 탭형 요약 화면은 축소하고, 실제 사용자 흐름과 가까운 route/section 구성으로 정리한다.

## 2차

### 작업내용

- 실제 제품과 데모 간 불일치 항목을 구조/상태/카피/CTA 기준으로 분해한다.
- 아래 우선순위로 차수별 구현을 자른다.
  1. 정보 구조 일치
  2. 주요 레이아웃/패널 일치
  3. CTA/네비게이션 일치
  4. mock 데이터 주입 안정화
  5. 시각 polish

### 논의 필요

- 데모 환경에서 어느 수준까지 상호작용을 허용할지(읽기 전용, 부분 상호작용, 거의 실사용 수준) 결정이 필요하다.

### 선택지

1. 읽기 중심 seeded demo
2. 핵심 CTA만 동작하는 반상호작용 demo
3. 실제 기능 대부분을 mock backend로 구동하는 풀인터랙션 demo

### 추천

- 먼저 2번으로 간다.
- 사용자는 “실제 제품을 보는 감각”이 중요하므로, 탭 전환/선택/모달 정도는 동작하고 저장성 작업은 제한하는 방향이 적절하다.
