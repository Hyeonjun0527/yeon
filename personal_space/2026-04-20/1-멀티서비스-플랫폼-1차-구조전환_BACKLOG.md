# 멀티서비스 플랫폼 1차 구조전환 BACKLOG

## 작업 목표

- 단일 배포는 유지한다.
- `yeon.world`를 포털 셸로 재정의한다.
- 상담 서비스는 `yeon.world/counseling-service` 하위 경로를 기준으로 동작하도록 구조를 바꾼다.
- 카카오/구글 로그인은 루트 플랫폼이 소유하되, 로그인 후 상담 서비스로 진입할 수 있게 맞춘다.
- 이후 `typing-service`, `beauty-ranking-service` 같은 새 서비스를 같은 방식으로 붙일 수 있는 경로/서비스 레지스트리 기반을 만든다.

## 차수 1. 플랫폼 셸과 서비스 경로 규약 도입

### 작업내용

- 루트 `/`를 포털 셸 진입점으로 유지한다.
- 서비스 경로 규약을 `/<service-slug>` 형태로 정한다.
- 1차 서비스 레지스트리 상수 또는 설정 파일을 도입한다.
- 상담 서비스의 앱 base path를 `/counseling-service`로 맞출 수 있게 라우팅 추상화를 확장한다.
- 현재 `/home` 중심 화면 진입을 `/counseling-service` 기준으로 이동한다.

### 논의 필요

- `/home`를 완전히 제거할지, 임시 호환 리다이렉트를 둘지
- 상담 서비스 내부 실제 메인 경로를 `/counseling-service`로 둘지 `/counseling-service/home`로 둘지

### 선택지

- 선택지 A: `/home`를 바로 없애고 `/counseling-service`를 메인으로 사용
- 선택지 B: 내부는 `/home`를 유지하고 외부만 `/counseling-service`로 매핑

### 추천

- 선택지 A
- 외부 URL과 내부 source of truth를 다르게 두면 이후 인증, 네비게이션, deep link 정리가 더 어려워진다.

### 사용자 방향

## 차수 2. 중앙 인증 소유권 정리

### 작업내용

- 카카오/구글 로그인 시작점은 루트 플랫폼이 소유한다.
- 로그인 후 기본 redirect를 상담 서비스 경로로 바꾼다.
- `next` 파라미터, 에러 페이지, 세션 cleanup 경로가 새 서비스 경로를 기준으로 동작하도록 맞춘다.
- 서비스별 익명/로그인 정책을 선언할 최소 구조를 만든다.

### 논의 필요

- 루트 로그인 UI를 포털 전용으로 둘지, 상담 전용 CTA를 같이 둘지
- 향후 익명 서비스 추가 시 로그인 유도 패턴을 어디서 제어할지

### 선택지

- 선택지 A: 로그인 UI는 루트 포털만 소유
- 선택지 B: 각 서비스도 자체 로그인 CTA를 가짐

### 추천

- 선택지 A
- 인증 소유권이 루트에 있다는 원칙을 유지하는 편이 장기적으로 덜 흔들린다.

### 사용자 방향

## 차수 3. 서비스 레지스트리와 포털 확장 포인트 도입

### 작업내용

- 서비스 목록/메타데이터를 공통 registry로 정리한다.
- 포털에서 각 서비스를 링크할 수 있게 기본 노출 구조를 만든다.
- 서비스별 접근 정책(`anonymous`, `auth-required`, `mixed`) 선언을 registry에 포함한다.
- 아직 구현되지 않은 서비스도 placeholder로 등록 가능한 구조를 만든다.

### 논의 필요

- registry를 앱 내부에 둘지 shared package로 올릴지
- 미구현 서비스 노출 방식을 `준비중`으로 할지 감출지

### 선택지

- 선택지 A: 현재는 `apps/web` 내부에 registry를 둔다
- 선택지 B: 곧바로 `packages/domain` 또는 새 shared package로 올린다

### 추천

- 선택지 A
- 아직 웹 앱 하나만 소비하므로 shared package 승격은 이르다.

### 사용자 방향

## 차수 4. 호환 리다이렉트와 검증

### 작업내용

- 기존 `/home`, legacy 상담 진입 경로를 새 경로로 안전하게 리다이렉트한다.
- 네비게이션, 링크 생성, 로그인 후 이동, 에러 복귀 경로를 검증한다.
- lint, format, typecheck, build를 통과시킨다.

### 논의 필요

- 레거시 경로를 얼마나 오래 유지할지
- 외부 공유 링크나 북마크 호환 정책

### 선택지

- 선택지 A: 핵심 레거시 경로만 유지
- 선택지 B: 가능한 많은 레거시 경로를 오래 유지

### 추천

- 선택지 A
- 유지 비용이 큰 레거시 경로는 빨리 줄이고, 핵심 진입점만 호환시키는 편이 낫다.

### 사용자 방향
