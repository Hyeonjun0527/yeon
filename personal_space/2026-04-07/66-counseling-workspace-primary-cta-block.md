# 상담 기록 워크스페이스 1차 CTA 블록 재설계

## 목적

- [65-counseling-workspace-empty-state-shell.md](/home/osuma/coding_stuffs/yeon/personal_space/2026-04-07/65-counseling-workspace-empty-state-shell.md)에서 정의한 첫 진입 셸 안에 `무엇이 가장 먼저 보여야 하는지`를 CTA 단위로 고정한다.
- 이번 차수의 핵심은 `업로드/녹음`을 더 예쁘게 만드는 것이 아니라, `입력 폼보다 먼저 보이게 만드는 것`이다.
- 구현 단계에서는 이 문서를 기준으로 첫 진입 행동 블록, 선택 오디오 상태, 저장 직전 흐름을 재배치한다.

## 현재 문제

## 코드 기준

- 업로드/녹음 액션이 학생 정보, 제목, 상담 유형 입력 아래에 배치돼 첫 행동이 묻힌다.
  - `counseling-record-workspace.tsx:1568-1624`
- `파일 선택`, `브라우저 녹음`이 둘 다 일반 `secondaryButton` 스타일이라 화면의 주인공처럼 보이지 않는다.
  - `counseling-record-workspace.tsx:1631-1650`
  - `counseling-record-workspace.module.css:335-388`
- 오디오를 아직 고르지 않았을 때도 CTA 아래에 별도 점선 빈 상태 박스와 긴 설명문이 붙어 밀도를 올린다.
  - `counseling-record-workspace.tsx:1680-1693`
  - `counseling-record-workspace.module.css:401-433`
- `최대 nMB`, `접기` 같은 보조 요소가 CTA 헤더 우측에 붙어 있어 시선이 분산된다.
  - `counseling-record-workspace.tsx:1557-1566`
- 접힌 상태의 CTA도 설명이 길고 약해서 `여기서 바로 시작하면 된다`는 인상이 약하다.
  - `counseling-record-workspace.tsx:1730-1741`

## UX 판단

- 지금 구조는 `새 기록 생성 폼`이 먼저 보이고, `업로드/녹음`은 그 폼 안의 한 단계처럼 보인다.
- 사용자의 실제 첫 행동은 폼 작성이 아니라 `오디오 확보`다.
- 따라서 첫 블록의 역할은 `입력 수집`이 아니라 `행동 시작`이어야 한다.

## 최종 결정

## 선택

- `A. 업로드와 녹음을 동일 위계`로 고정한다.

## 이유

- 사용자 요구 자체가 두 행동을 모두 핵심 CTA로 보이게 하라는 방향이다.
- 한쪽만 primary로 강하게 만들면 다른 한쪽이 `보조 기능`처럼 밀려 보인다.
- 현재 문제는 `무엇이 더 중요하냐`보다 `둘 다 너무 약하게 보인다`에 가깝다.
- 따라서 `같은 크기, 같은 무게, 같은 시선 우선순위`로 두고, 실제 상태 진입 후에만 하나의 흐름을 좁히는 편이 맞다.

## 예외 처리

- 동일 위계를 유지하되, 좌우 순서는 `파일 업로드 -> 브라우저 녹음`으로 둔다.
- 이유는 업로드가 더 일반적인 시작점이기 때문이지, 더 높은 위계라는 뜻은 아니다.

## CTA 블록 역할 재정의

### 지금까지

- `상담 메타 입력을 포함한 생성 폼`

### 앞으로

- `새 기록을 시작하는 행동 런처`

## 포함해야 하는 것

- 섹션 제목
- 1줄 보조 문구
- `파일 업로드`
- `브라우저 녹음`
- 오디오가 선택된 뒤의 짧은 확인 영역
- 오디오가 준비된 뒤의 `기록 저장`

## 포함하면 안 되는 것

- 학생 이름, 학년/반, 상담 제목, 상담 유형의 상시 노출
- 긴 업로드/보관 설명
- 상단 수치 배지
- CTA 내부 점선 빈 박스
- `설명용 카드 안의 설명용 카드`

## 상태별 구조

## 1. 기본 idle 상태

### 사용자 질문

- `어디서 시작하지?`

### 화면 답변

- `파일 업로드`
- `브라우저 녹음`

### 구조

```text
새 기록 만들기
파일을 올리거나 바로 녹음해 시작합니다.

[ 파일 업로드 ]
오디오 파일에서 시작

[ 브라우저 녹음 ]
지금 바로 녹음 시작
```

### 규칙

- CTA 2개가 블록 안에서 가장 크고 먼저 보여야 한다.
- 두 CTA는 가로 2분할보다 세로 stack을 기본값으로 둔다.
- 이유는 좌측 패널 폭이 좁고, 모바일에서도 그대로 유지하기 쉽기 때문이다.
- 하단 빈 상태 박스는 없다.

## 2. 파일 선택 후 상태

### 사용자 질문

- `이 파일로 저장하면 되나?`

### 화면 답변

- `선택한 오디오 확인 -> 기록 저장`

### 구조

```text
새 기록 만들기
선택한 오디오를 확인한 뒤 저장합니다.

[ 선택한 오디오 카드 ]
- 파일명
- 길이 / 크기
- 오디오 미리듣기

[ 기록 저장 ]
다른 파일 선택
브라우저 녹음
```

### 규칙

- 이 상태에서는 `기록 저장`이 가장 강한 primary가 된다.
- 기존 2개 CTA는 완전히 사라지지 않되, 동일 크기 타일로 유지하지 않는다.
- 현재 흐름과 관계없는 행동은 `다른 파일 선택`, `브라우저 녹음` 수준의 약한 보조 액션으로 내린다.

## 3. 녹음 시작 후 상태

### 사용자 질문

- `녹음이 진행 중인지, 멈추면 저장할 수 있는지`

### 화면 답변

- `녹음 진행 상태 -> 중지 -> 저장`

### 구조

```text
새 기록 만들기
녹음을 마치면 바로 저장할 수 있습니다.

[ 녹음 상태 블록 ]
- 현재 녹음 중
- 경과 시간
- 중지 버튼

[ 기록 저장 ] 또는 [ 중지 후 저장 준비 ]
파일 업로드
```

### 규칙

- 녹음 중에는 `브라우저 녹음` 타일이 계속 남아 있지 않는다.
- 하나의 상태 블록으로 전환해 흐름을 단순하게 보여준다.
- 녹음 마이크로플로우의 세부 표현은 12차에서 더 다듬되, 8차부터 `타일 유지 방식`은 금지한다.

## 레이아웃 규칙

## 구조 방향

- 좌측 생성 패널 상단에는 `섹션 제목 + 1줄 설명 + 대형 CTA stack`만 온다.
- CTA 아래에 바로 오디오 확인 영역이 이어진다.
- 추가 입력은 CTA 블록 바깥의 접힌 섹션으로 밀어낸다.

## 배치 원칙

- CTA 2개는 같은 높이와 같은 패딩을 쓴다.
- 각 CTA는 `아이콘 / 제목 / 1줄 설명` 구조를 가진다.
- 버튼 안에 긴 상태 문장을 넣지 않는다.
- `justify-between`으로 라벨과 보조 설명을 과하게 벌리지 않는다.

## 시각 규칙

### 1. CTA는 버튼이 아니라 작은 액션 타일처럼 보여야 한다

- 일반 `secondaryButton`보다 크다
- 최소 높이 `72-84px`
- 아이콘 `18-20px`
- 제목 `15-16px / 700`
- 설명 `12-13px / muted`

### 2. 강한 색은 전체 칠이 아니라 선택/hover에만 쓴다

- idle 상태에서는 두 CTA 모두 `surface` 기반
- hover 또는 focus에서만 `accent` 보더나 `accent-soft` 배경 사용
- 선택된 흐름 진입 후에만 해당 블록이 약하게 활성화된다

### 3. CTA 주변은 박스보다 여백으로 분리한다

- CTA 아래 별도 점선 empty card 금지
- 설명문은 CTA 위 1줄만 허용
- 상단 우측의 용량 제한 pill은 제거하거나 하단 소형 캡션으로 이동

## 이번 차수의 완료 판단

- 첫 진입 시 좌측 패널에서 가장 먼저 읽히는 요소가 `파일 업로드`, `브라우저 녹음`인지
- 학생 정보 입력 없이도 생성 시작이 가능하게 설계됐는지
- 선택 오디오가 생긴 뒤에는 `기록 저장`이 다음 주인공으로 올라오는지
- 긴 설명과 점선 빈 박스가 CTA 블록 안에서 사라졌는지
