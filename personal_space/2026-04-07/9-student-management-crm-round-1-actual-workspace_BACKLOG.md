# 학생관리 CRM 라운드 1 실제 워크스페이스 전환 백로그

## 1차

### 작업내용

- 홈(`/`)을 설명형 랜딩이 아니라 교강사 학생관리 워크스페이스로 전환한다.
- 기존 공모전 소개 화면은 별도 경로로 옮겨 보존하고, 홈에서는 학생 큐, 상세 패널, 오늘 액션 보드를 한 화면에서 닫는다.
- `instructor-workspace` 읽기 계약과 mock source를 연결해 실제 작업형 UI가 공용 계약을 바라보게 만든다.

### 논의 필요

- 홈에서 공모전 설명 섹션을 완전히 제거할지, 보조 진입으로 남길지 판단이 필요하다.

### 선택지

- A. 홈은 워크스페이스 우선, 공모전 화면은 별도 경로로 분리
- B. 홈에서 설명형 랜딩과 워크스페이스를 함께 노출

### 추천

- A. 사용자가 요구한 "실제 제품" 인상이 더 강하고 첫 스캔이 빨라진다.

## 2차

### 작업내용

- 학생 검색, 세그먼트 필터, 학생 선택, 액션 상태 변경, 메시지 초안 확인 흐름을 client feature로 구현한다.
- 모바일 375px부터 데스크톱까지 우선순위가 유지되도록 반응형 레이아웃과 포커스 상태를 정리한다.

### 논의 필요

- 메모 입력을 영속 저장까지 연결할지, 이번 차수는 브라우저 세션 기반 draft 경험까지만 제공할지 정해야 한다.

### 선택지

- A. 이번 차수는 읽기 + 조작형 draft UI까지 구현
- B. 바로 쓰기 API까지 같이 연결

### 추천

- A. 현재 저장소의 round 1 목적은 실제 제품 흐름을 먼저 닫는 것이고, 쓰기 영속화는 다음 landing 단위로 분리하는 편이 안전하다.

### 사용자 방향

## 3차

### 작업내용

- `pnpm lint`, `pnpm prettier:fix`, `pnpm typecheck`, `pnpm --filter @yeon/web build` 순서로 검증한다.
- 검증 후 내 변경만 add/commit/push하고 PR 생성 및 merge까지 진행한다.

### 논의 필요

- 이번 차수에서 Playwright E2E까지 추가할지 여부

### 선택지

- A. 코드 검증과 빌드 green을 우선 확보
- B. 로컬 E2E까지 같이 추가

### 추천

- A. 홈 전환과 상호작용 UI landing을 먼저 안전하게 끝내고, E2E는 다음 라운드로 분리한다.
