# 상담 기록 워크스페이스 첫 진입 빈 상태 셸 재구성 설계

## 목적

- 현재 워크스페이스는 데이터가 없거나 선택 전일 때 `좌측 업로드 빈 상태 + 중앙 빈 상태 + 우측 AI 빈 상태`가 동시에 떠서 미완성처럼 보인다.
- 이번 차수에서는 빈 상태를 `여러 박스`가 아니라 `하나의 행동 중심 셸`로 바꾼다.
- 구현 단계에서는 이 문서를 기준으로 첫 진입, 기록 0건, 기록 있음 but 미선택 상태를 분리한다.

## 코드 기준

- 오디오가 없으면 좌측 업로드 카드 안에 또 별도 빈 상태 박스가 뜬다.

- 기록이 선택되지 않으면 중앙에 큰 빈 상태 박스가 뜬다.
  - `counseling-record-workspace.tsx:2368-2377`
  - `counseling-record-workspace.module.css:680-691`
- 기록이 선택되지 않아도 우측 AI 패널 칼럼과 빈 상태 박스가 유지된다.
  - `counseling-record-workspace.tsx:2381-2493`
  - `counseling-record-workspace.module.css:155-165`
- 기록이 0건이면 업로드 패널을 자동으로 열어 두므로 첫 진입부터 입력 덩어리가 커진다.
  - `counseling-record-workspace.tsx:934-938`

## UX 기준

- 빈 상태에서는 사용자가 `무엇을 클릭해야 하는지`만 알면 된다.
- 첫 인상은 `여기서 업로드하면 되는구나`여야지, `아직 준비 안 된 화면 같네`가 되면 안 된다.
- 기록이 없을수록 화면은 더 단순해져야 한다.

## 최종 목표

### 한 줄 정의

- `빈 상태일수록 더 적게 보이고, 더 크게 행동을 보여주는 셸`

### 핵심 원칙

- 박스 수를 줄인다
- 설명문을 줄인다
- 선택 전 AI 패널은 제거한다
- 첫 행동은 업로드/녹음 두 개만 보여 준다
- 기록이 생기면 그때부터 탐색 구조를 확장한다

## 구분해야 할 빈 상태 3종

## 1. 완전 첫 진입 / 기록 0건

### 상황

- 처음 들어왔고 기록이 없다

### 사용자 질문

- `어떻게 시작하지?`

### 화면 답변

- `파일 업로드` 또는 `브라우저 녹음`

### 구조

| 영역 | 구성 |
| --- | --- |
| 상단 | 슬림 헤더 |
| 좌측 | 대형 CTA 2개 |
| 중앙 | 짧은 한 덩어리 안내 |
| 우측 | 없음 |

### 없애야 하는 것

- 학생 정보 폼
- 업로드 카드 내부 빈 상태 박스
- 중앙 대형 박스형 empty panel
- 우측 AI empty panel
- 검색창, 필터

## 2. 생성 중 / 저장 전

### 상황

- 오디오를 고르는 중이거나 녹음 중이다

### 사용자 질문

- `이 오디오를 저장하면 되나?`

### 화면 답변

- `오디오 확인 -> 저장`

### 구조

| 영역 | 구성 |
| --- | --- |
| 상단 | 슬림 헤더 |
| 좌측 | CTA + 선택 오디오 + 저장 버튼 + 접힌 추가 정보 |
| 중앙 | 짧은 진행 안내 또는 최근 기록 1~2개 |
| 우측 | 없음 |

### 없애야 하는 것

- 우측 AI
- 상세/원문 뷰어
- 큰 빈 상태 패널 반복

## 3. 기록 있음 / 아직 미선택

### 상황

- 기록은 있지만 아직 아무 것도 열지 않았다

### 사용자 질문

- `어느 기록을 열지? 아니면 새로 만들지?`

### 화면 답변

- 좌측에서 기록 선택
- 또는 새 기록 만들기

### 구조

| 영역 | 구성 |
| --- | --- |
| 상단 | 슬림 헤더 |
| 좌측 상단 | 접힌 `새 기록 만들기` CTA |
| 좌측 본문 | 기록 리스트 |
| 중앙 | 선택 유도 안내 + 최근/방금 생성한 기록 빠른 액션 |
| 우측 | 없음 |

### 없애야 하는 것

- 우측 AI 빈 상태
- 중앙 거대 dashed empty panel
- 상단 수치 메트릭

## 최종 셸 구조

```text
상단: 얇은 제목줄

본문:
  좌측: 시작 행동 또는 기록 탐색
  중앙: 한 덩어리 안내 또는 빠른 선택
  우측: 선택 전에는 없음
```

## 시각 방향

### 1. 좌측은 시작점처럼 보여야 한다

- CTA 블록이 가장 크다
- 설명보다 버튼이 먼저 읽힌다
- `업로드 카드 안의 빈 박스` 구조를 없앤다

### 2. 중앙은 보조 캔버스처럼 보여야 한다

- 거대한 비어 있는 판넬이 아니라
- 짧은 제목 + 1줄 설명 + 보조 액션 정도의 가벼운 구조

### 3. 우측은 선택 전 완전히 빠진다

- 자리 차지만 해도 밀도가 올라간다
- AI는 선택 후에만 등장해야 흐름이 맞다

## 분기 규칙

### `records.length === 0`

- `CreateRecordPanel`은 `expanded` 상태로 시작 가능
- 하지만 학생 정보 필드까지 다 펼치지 않는다
- 중앙은 `첫 기록을 만들면 여기서 원문을 확인합니다` 정도의 단일 안내만 표시

### `records.length > 0 && !selectedRecord`

- `CreateRecordPanel`은 접힌 상태 기본
- `RecordList`는 노출
- 중앙에는 `기록을 선택하세요`와 빠른 선택/방금 만든 기록만 표시

### `selectedRecord`

- 이때만 기존 3영역 구조 완성

## 권장 스타일

- 배경은 투명 또는 아주 약한 표면색
- 보더는 없어도 된다
- 꼭 보더가 필요하면 아래쪽 구분선 정도만
- 빈 상태 높이는 `콘텐츠에 맞는 높이`로 유지하고, 큰 고정 높이 박스를 만들지 않는다

## 레이아웃 규칙

- 첫 진입에서는 좌측 CTA와 중앙 안내가 같은 높이 경쟁을 하지 않는다
- 중앙 안내는 좌측보다 시각적 무게가 낮아야 한다
- 모바일에서는 `CTA -> 안내 -> 목록` 순서로 자연스럽게 세로 배치한다

## 구현 체크리스트

1. 선택 전 AI 패널을 아예 렌더하지 않기
2. 중앙 빈 상태를 하나의 행동 유도 셸로 바꾸기
3. 업로드 패널 내부 빈 상태 설명문 축약하기
4. 기록 0건 상태와 기록 있음 미선택 상태를 분리하기
5. `records.length === 0`일 때 자동으로 큰 입력 폼이 과하게 펼쳐지지 않게 조정하기

## 완료 기준

- 기록이 없을수록 화면이 더 단순해 보여야 한다
- 첫 진입에서 사용자는 3초 안에 `업로드/녹음` 행동을 파악할 수 있어야 한다
- 선택 전 화면에서 우측 큰 빈 박스가 없어야 한다
- 중앙 영역은 `미완성 박스`가 아니라 `다음 행동 안내`처럼 보여야 한다

## 차수 8 입력값

- 차수 8에서는 이 빈 상태 셸 기준을 유지한 채, 좌측 1차 CTA 블록 자체를 더 크게 재설계한다.
