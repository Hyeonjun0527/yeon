# 상담 기록 워크스페이스 생성 폼 점진적 노출 구조 설계

## 목적

- [66-counseling-workspace-primary-cta-block.md](/home/osuma/coding_stuffs/yeon/personal_space/2026-04-07/66-counseling-workspace-primary-cta-block.md)에서 정의한 1차 CTA 뒤에 어떤 입력이 언제 나타나야 하는지 고정한다.
- 이번 차수의 핵심은 `입력 항목을 없애는 것`이 아니라, `입력이 필요한 순간까지 숨기는 것`이다.
- 구현 단계에서는 이 문서를 기준으로 생성 폼을 `첫 행동 -> 최소 확인 -> 추가 정보` 3단계로 재배치한다.

## 현재 문제

## 코드 기준

- 업로드 패널이 열리면 학생 이름, 학년/반, 상담 제목, 상담 유형이 모두 즉시 노출된다.
  - `counseling-record-workspace.tsx:1577-1626`
- 저장 전에 실제로 검증하는 값은 `학생 이름`, `상담 제목`, `오디오 선택 여부`인데, 시각적으로는 모든 필드가 같은 무게로 보인다.
  - `counseling-record-workspace.tsx:1336-1363`
- 오디오를 고르면 이미 `sessionTitle`은 파일명 기준으로 자동 채워지는데도 제목 입력이 계속 전면에 노출된다.
  - `counseling-record-workspace.tsx:1069-1095`
- 저장 성공 후 `sessionTitle`만 초기화되고 `studentName`, `gradeLabel`, `counselingType`은 남아서, 나중에 필드를 숨기면 이전 값이 보이지 않게 이어질 위험이 있다.
  - `counseling-record-workspace.tsx:1429-1432`

## UX 판단

- 첫 화면에서 모든 입력을 다 보여주면 사용자는 `무엇을 먼저 해야 하는지`보다 `채워야 할 게 많다`를 먼저 느낀다.
- 반대로 필수값을 너무 숨기면 저장 직전에 흐름이 막힌다.
- 따라서 `처음에는 CTA만`, `오디오가 준비되면 최소 확인`, `그 외는 추가 정보`라는 단계가 필요하다.

## 계약 제약

## 현재 소스 오브 트루스

- 클라이언트 저장 시 `studentName`, `sessionTitle`은 빈 값이면 막힌다.
  - `counseling-record-workspace.tsx:1348-1363`
- 서버 서비스도 `학생 이름`, `상담 제목`을 required로 처리한다.
  - `counseling-records-service.ts:1084-1088`
- DB 역시 `student_name`, `session_title`, `counseling_type`은 not null이다.
  - `schema/counseling-records.ts:17-20`

## 의미

- 이번 차수는 `필수값 제거`가 아니라 `필수값 노출 시점 재배치`로 가야 한다.
- 특히 `학생 이름`을 완전히 숨긴 채 자동 placeholder로 저장하는 방향은 기록 신뢰도를 떨어뜨리므로 추천하지 않는다.

## 최종 결정

## 선택

- 백로그 추천안 `B. 파일/녹음 완료 후 자동 생성하고 수동 수정은 접힌 영역에 배치`를 채택한다.

## 구체화

- `상담 제목`은 최소 입력에서 제거한다.
- `상담 제목`은 오디오 선택 시 자동 생성하고, 필요할 때만 `추가 정보`에서 수정한다.
- `학생 이름`은 첫 진입에서는 숨기되, 오디오가 준비된 뒤 `최소 확인 정보`로 올라온다.
- `학년/반`, `상담 유형`은 `추가 정보`에 둔다.

## 이유

- 제목은 현재 코드상 이미 자동 생성 기반이 있어 접어도 무리가 적다.
- 학생 이름은 저장 신뢰도와 리스트 탐색 품질에 직접 영향을 주므로 완전 숨김보다 `저장 직전 최소 노출`이 맞다.
- 학년/반, 상담 유형은 생성 품질 보조값이지 첫 저장을 막을 정도의 1차 정보가 아니다.

## 최종 단계 구조

## 1. 첫 행동 단계

### 노출

- `파일 업로드`
- `브라우저 녹음`

### 숨김

- 학생 이름
- 상담 제목
- 학년/반
- 상담 유형

### 목적

- 사용자가 일단 오디오를 확보하게 만든다.

## 2. 최소 확인 단계

### 진입 조건

- 파일이 선택되었거나 녹음이 종료되어 저장 가능한 오디오가 생김

### 노출

- 선택한 오디오 카드
- `학생 이름` 1개 필드
- `기록 저장`
- `추가 정보` 토글

### 숨김

- 상담 제목 입력
- 학년/반 입력
- 상담 유형 선택

### 목적

- 기록 생성 전에 정말 필요한 최소 확인만 받는다.

## 3. 추가 정보 단계

### 노출

- 상담 제목
- 학년/반
- 상담 유형

### 성격

- 선택 입력
- 기본 저장 흐름을 막지 않는 2차 정보

## 필드 분류

| 필드 | 노출 시점 | 성격 | 비고 |
| --- | --- | --- | --- |
| 오디오 선택 | 첫 행동 | 필수 | CTA 단계 |
| 학생 이름 | 최소 확인 | 필수 | 저장 전 보여야 함 |
| 상담 제목 | 추가 정보 | 필수 값이지만 자동 생성 | 접힌 상태 기본 |
| 학년/반 | 추가 정보 | 선택 | 보조 메타 |
| 상담 유형 | 추가 정보 | 선택에 가까운 기본값 | 기본값 유지 가능 |

## 상담 제목 처리 규칙

## 자동 생성 원칙

- 업로드 파일이면 확장자를 제거한 파일명을 기본 제목으로 쓴다.
- 현재 코드에도 이 동작이 이미 들어 있다.
  - `counseling-record-workspace.tsx:1092-1094`
- 브라우저 녹음이면 현재의 기계적인 파일명 대신 사람이 읽을 수 있는 기본 제목으로 바꾸는 것이 좋다.

## 권장 기본 제목

| 상황 | 권장 제목 |
| --- | --- |
| 업로드 파일 | 파일명 기반 자동 제목 |
| 브라우저 녹음 | `브라우저 녹음 2026.04.07 14:30` 형태 |

## 수정 방식

- 제목 입력은 기본 접힘 상태의 `추가 정보` 안에 둔다.
- 사용자가 제목 품질을 신경 쓰는 경우에만 열어 직접 수정한다.

## 학생 이름 처리 규칙

## 왜 최소 확인 단계에 남겨야 하는가

- 학생 이름은 리스트 탐색, 학생 누적 기록, 후속 열람 품질에 직접 연결된다.
- 현재 제품 방향에서도 학생별 누적 기록은 신뢰성의 일부다.
- 따라서 `완전 자동 placeholder 저장`은 비추천이다.

## 노출 방식

- 오디오 카드 바로 아래에 단일 필드로 둔다.
- 첫 진입에서는 보이지 않다가 오디오가 생긴 순간 나타난다.
- 저장 버튼과 가까이 붙여 `지금 확인하면 되는 값`처럼 읽히게 한다.

## 비추천

- 첫 진입부터 학생 이름을 계속 노출하는 방식
- 학생 이름을 `추가 정보` 안으로 완전히 숨기는 방식
- 사용자가 못 본 이전 학생 이름이 다음 기록에 그대로 남는 방식

## 추가 정보 섹션 규칙

## 포함 필드

- 상담 제목
- 학년/반
- 상담 유형

## 노출 원칙

- 기본은 닫힘
- 한 줄 요약 라벨만 먼저 보인다
- 저장은 펼치지 않아도 가능하다

## 라벨 방향

- 이번 차수에서는 섹션 이름을 우선 `추가 정보`로 가정한다.
- 세부 라벨 확정은 10차에서 한다.

## 상태 전이 규칙

## 오디오 선택 전

- 폼 입력 자체를 렌더하지 않거나, 최소한 사용자 눈에는 보이지 않게 둔다.

## 오디오 선택 후

- `학생 이름` 필드가 열린다.
- `추가 정보` 디스클로저가 등장한다.
- 저장 버튼이 활성 구조 안으로 들어온다.

## 저장 성공 후

- 최소 입력과 추가 입력은 다음 기록 생성을 위해 다시 접힌 기본 상태로 돌아가야 한다.
- 특히 hidden 상태의 `학생 이름`, `학년/반`이 남아 있지 않게 정리해야 한다.
- 현재처럼 `sessionTitle`만 초기화하는 방식은 점진적 노출 구조와 충돌한다.

## 저장 실패 시

- 현재처럼 인라인 피드백은 유지한다.
- 단, 오류가 `학생 이름 누락`이면 해당 필드를 자동으로 보이게 하고 포커스를 이동해야 한다.

## 마크업 방향

## 스타일 규칙

### 최소 확인 필드

- CTA 타일보다 약하지만, 추가 정보보다 강해야 한다.
- 단일 입력으로 보이게 하고 `fieldGrid` 2열 기본값은 쓰지 않는다.
- 저장 버튼과 가까운 세로 흐름으로 배치한다.

### 추가 정보

- 별도 큰 카드 금지
- 약한 보더 또는 텍스트 토글 정도로만 시작
- 펼쳤을 때도 내부 필드 수가 많아 보이지 않게 간격을 줄인다

## 구현 메모

- 현재 `applySelectedAudioFile`의 제목 자동 채우기 로직은 그대로 활용 가능하다.
- 저장 검증 로직은 `학생 이름`과 `오디오` 중심으로 재배치하고, 제목 누락 검증은 자동 제목 생성이 보장되도록 먼저 정리해야 한다.
- 성공 후 폼 reset 정책은 `점진적 노출에 맞는 초기 상태`로 다시 설계해야 한다.

## 이번 차수의 완료 판단

- 첫 진입에서는 입력 필드가 보이지 않고 CTA만 보이는지
- 오디오 선택 후에만 `학생 이름`과 `저장`이 나타나는지
- `상담 제목`이 기본 접힘 상태로 내려갔는지
- hidden 필드의 이전 값이 다음 기록에 조용히 재사용되지 않도록 설계됐는지
