## 1차: 스페이스 개념 도입 + 용어 교정

### 작업내용

"년도 탭"을 **스페이스(Space)** 로 교체한다.
스페이스는 수강생을 그룹핑하는 사용자 정의 단위다.

#### 스페이스란?

- 운영자가 직접 만들고 이름을 짓는 수강생 관리 단위
- 예시: "백엔드 3기", "디자인 스쿨 2026 상반기", "데이터 분석 입문", "1:1 멘토링"
- 하나의 수강생이 여러 스페이스에 소속 가능 (N:M)
- 스페이스별로 독립적인 수강생 목록, 상담 기록 필터, 통계 제공

#### UI 변경

현재 mockdata-v2의 년도 탭 영역을 스페이스 셀렉터로 교체한다.

```
Before: [2026] [2025] [2024] [전체]
After:  [전체] [백엔드 3기] [디자인 2기] [1:1 멘토링] [+ 스페이스 추가]
```

#### 스페이스 CRUD

- **생성**: 이름, 설명, 컬러/아이콘(선택), 시작일~종료일(선택)
- **수정**: 이름, 설명, 기간 변경
- **삭제**: 소속 수강생 연결만 해제, 수강생 데이터는 유지
- **아카이브**: 종료된 스페이스는 아카이브로 이동 (숨김 처리, 조회 가능)

### 논의 필요

1. 스페이스를 폴더처럼 중첩 가능하게 할지 (예: "백엔드" > "3기") 아니면 플랫하게만 할지?
2. 기본 스페이스("전체")는 시스템이 자동 생성하고 삭제 불가로 할지?
3. 스페이스 간 수강생 이동/복사 UI가 필요한지?

### 선택지

| # | 항목 | A | B | 추천 |
|---|------|---|---|------|
| 1 | 구조 | 플랫 (태그처럼) | 중첩 (폴더처럼) | **A** — 1차에서는 플랫, 복잡도 최소화 |
| 2 | 기본 스페이스 | "전체" 고정 | 없음 (첫 스페이스 직접 생성) | **A** — "전체"는 항상 존재, 필터 없는 뷰 |
| 3 | 이동/복사 | 수강생 카드에서 스페이스 태그 추가/제거 | 별도 이동 UI | **A** — 태그처럼 다중 소속이 자연스러움 |

### 추천

플랫 구조, "전체" 기본 스페이스, 태그 방식 다중 소속. 1차에서는 단순하게 가고 필요 시 중첩 구조 추가.

## 2차: 수강생 엔티티 재설계

### 작업내용

부트캠프 맥락에 맞게 수강생 데이터 모델을 재설계한다.

#### 핵심 엔티티

| 엔티티 | 설명 | 주요 필드 |
|--------|------|-----------|
| **members** | 수강생 프로필 | id, name, email, phone, track, status(active/completed/dropped), enrolledAt, avatarUrl, memo |
| **spaces** | 스페이스 (기수/프로그램) | id, name, description, color, startDate, endDate, status(active/archived), createdByUserId |
| **space_members** | 스페이스-수강생 연결 (N:M) | id, spaceId, memberId, joinedAt, role(member/mentor) |
| **member_tags** | 수강생 태그 | id, name, color, createdByUserId |
| **member_tag_assignments** | 태그-수강생 연결 | memberId, tagId |

#### 이전 모델과의 차이

- `student_guardians` 제거 (성인 대상, 보호자 불필요)
- `classes` → `spaces` (반 → 스페이스, 더 유연한 그룹핑)
- `class_enrollments` → `space_members` (수강 관계 + role 추가)
- `students` → `members` (학생 → 수강생, 필드 경량화)
- `grade`, `school`, `birthDate`, `gender` 제거 → `track`, `email` 추가

### 논의 필요

1. 수강생 상태를 `active/completed/dropped` 3가지로 할지, `paused(휴강)` 추가할지?
2. 멘토도 같은 `members` 테이블에 role로 구분할지, 별도 테이블로 할지?
3. 수강생 자동 등록(상담 AI 추출) 흐름은 유지할지?

### 선택지

| # | 선택지 | 장점 | 단점 |
|---|--------|------|------|
| A | 멘토를 members에 role로 구분 | 단순, 스페이스 내 통합 관리 | 멘토 전용 기능 확장 시 분리 필요 |
| B | 멘토를 별도 테이블로 분리 | 역할 분리 명확 | 스페이스 소속 로직 복잡 |

### 추천

**A** — 멘토도 스페이스의 구성원이므로 `space_members.role`로 구분. 1차에서 충분.

## UI/UX 디자인 원칙 (부트캠프 맥락)

### 부트캠프 운영 특화

1. **스페이스 중심 관리**: 기수/프로그램별로 수강생을 묶어 관리. 동시에 여러 프로그램 운영 가능.
2. **빠른 전환**: 멘토는 수업/멘토링 사이 짧은 시간에 확인. 3클릭 이내 정보 도달.
3. **벌크 작업**: 기수 전환 시 대규모 수강생 이동, 수료 처리 등 일괄 작업 필수.
4. **모바일 대응**: 태블릿/폰에서도 수강생 확인, 멘토링 기록 접근 가능.

### 시각 디자인

1. 깔끔한 카드 UI, 넉넉한 여백
2. 컬러 태그 + 스페이스 뱃지로 시각적 구분
3. 아바타 이니셜 + 파스텔 배경 자동 생성
4. 상태 뱃지: 수강중(초록), 수료(파랑), 중도포기(회색), 휴강(노랑)
5. Framer Motion 마이크로 인터랙션
